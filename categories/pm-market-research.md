# 시장 조사 (Market Research)

## 카테고리 소개

제품을 만들기 전에 "누구를 위한 제품인가"를 명확히 하는 것이 시장 조사의 핵심이다. 아무리 뛰어난 기술을 가지고 있어도, 사용자가 실제로 겪는 문제를 해결하지 않으면 제품은 실패한다. 시장 조사 카테고리는 사용자를 이해하고, 시장 기회를 발견하고, 경쟁 환경을 파악하는 데 필요한 스킬 7개를 제공한다.

비개발자 PO에게 이 카테고리가 특히 중요한 이유는, 개발 이전 단계에서 "만들어야 할 것"과 "만들지 말아야 할 것"을 구분하는 판단 근거를 제공하기 때문이다. PRD에 "1인 개발자"라고만 적혀 있다면, 그 안에 어떤 세그먼트가 있는지, 각 세그먼트가 어떤 여정을 거치는지, 경쟁자 대비 어디서 이길 수 있는지를 파악하는 것이 첫걸음이다.

이 카테고리의 스킬들은 데이터 기반 의사결정의 출발점이 된다. 설문, 인터뷰, 리뷰, 사용 로그 등 다양한 소스에서 패턴을 발견하고, 이를 구조화된 전략 문서로 만들어준다.

## 이 카테고리의 스킬 한눈에 보기

| 스킬명 | 한 줄 설명 | 난이도 | 예상 소요 시간 |
|--------|-----------|--------|--------------|
| 사용자 페르소나 (user-personas) | 리서치 데이터로 3개 페르소나를 JTBD/고통/이득과 함께 생성 | 초급 | 15-20분 |
| 시장 세그먼트 (market-segments) | 3-5개 고객 세그먼트를 인구통계/JTBD/제품 적합도로 분석 | 초급 | 15-20분 |
| 사용자 세분화 (user-segmentation) | 피드백 데이터에서 행동/JTBD/니즈 기반으로 3개 이상 세그먼트 식별 | 중급 | 25-35분 |
| 고객 여정 맵 (customer-journey-map) | 인지~옹호까지 단계별 터치포인트/감정/기회 매핑 | 초급 | 20-30분 |
| 시장 규모 추정 (market-sizing) | TAM/SAM/SOM을 Top-down/Bottom-up 방식으로 추정 | 중급 | 30-40분 |
| 경쟁 분석 (competitor-analysis) | 5개 직접 경쟁자의 강점/약점/차별화 기회 분석 | 초급 | 20-30분 |
| 감성 분석 (sentiment-analysis) | 대규모 사용자 피드백에서 세그먼트별 만족도와 개선 기회 도출 | 고급 | 35-45분 |

## 추천 실행 순서

시장 조사 스킬은 "넓은 시각 -> 좁은 초점"의 순서로 실행하는 것이 효과적이다.

1. **시장 세그먼트** (market-segments) -- 전체 시장의 윤곽을 잡는다. 3-5개 세그먼트를 식별해 "어디에 기회가 있는가"를 파악한다.
2. **사용자 페르소나** (user-personas) -- 가장 유망한 세그먼트 내에서 구체적인 사용자 프로필을 만든다.
3. **경쟁 분석** (competitor-analysis) -- 타겟 세그먼트에서 경쟁자가 어떻게 포지셔닝하고 있는지 파악한다.
4. **고객 여정 맵** (customer-journey-map) -- 타겟 페르소나가 제품을 발견하고 사용하는 전체 경험을 매핑한다.
5. **시장 규모 추정** (market-sizing) -- 선택한 세그먼트의 시장 크기를 숫자로 추정한다.
6. **사용자 세분화** (user-segmentation) -- 실제 사용자 데이터가 쌓인 후, 행동 기반으로 더 정밀하게 세분화한다.
7. **감성 분석** (sentiment-analysis) -- 사용자 리뷰/피드백이 충분히 쌓인 후, 세그먼트별 만족도를 정량적으로 측정한다.

> 1-5번은 제품 출시 전/초기에 실행하고, 6-7번은 사용자 데이터가 쌓인 후에 실행한다.

## 관련 카테고리

- **Go-to-Market**: 시장 조사 결과를 바탕으로 "어떻게 시장에 진입할 것인가"를 설계한다. 특히 `market-segments` -> `beachhead-segment` -> `gtm-strategy`의 흐름이 자연스럽다.
- **Marketing & Growth**: 시장 조사에서 발견한 가치 제안을 마케팅 메시지로 전환한다. `competitor-analysis`의 차별화 포인트가 `positioning-ideas`와 `value-prop-statements`의 입력이 된다.
- **Data Analytics**: 정성적 시장 조사를 정량적 데이터로 보완한다. `customer-journey-map`의 퍼널 단계를 `sql-queries`로 측정하고, `cohort-analysis`로 세그먼트별 리텐션을 추적한다.

## 스킬 상세

### 1. 사용자 페르소나 (user-personas)
**난이도**: 초급
**예상 소요 시간**: 15-20분
**슬래시 명령어**: `/pm-market-research:user-personas`

#### 이 스킬은 언제 쓰나요?
- 설문/인터뷰 데이터에서 구체적인 사용자 프로필을 만들고 싶을 때
- "우리 사용자가 누구인지" 팀 내에서 공유할 수 있는 문서가 필요할 때
- 사용자 유형별로 제품 기능의 우선순위를 다르게 매기고 싶을 때

#### 실행 방법
```
/pm-market-research:user-personas SprintX - AI 에이전트 기반 스프린트 관리 SaaS.
타겟: 1인 개발자 및 소규모 팀.
기존 데이터: 사용자 인터뷰 3건, Waitlist 등록자 정보.
```

데이터 파일(CSV, 설문 응답, 인터뷰 기록)이 있다면 함께 제공하면 더 정확한 페르소나를 생성한다.

#### 산출물 예시
```
## 페르소나 1: 민수 (사이드 프로젝트 개발자)

**나이/역할**: 28세, 프론트엔드 개발자 (회사원 겸 사이드 프로젝트)
**회사 규모**: 개인

**Primary Job-to-be-Done**:
퇴근 후 제한된 시간에 사이드 프로젝트를 효율적으로 진행하고 싶다.
주 2-3회, 회당 2시간 집중 작업.

**Top 3 Pain Points**:
1. 매번 "오늘 뭐 했지, 다음에 뭘 하지"를 기억에 의존 (심각도: 높음)
2. AI 코딩 에이전트가 만든 코드를 검토할 체계가 없음 (심각도: 중간)
3. 진행 상황을 시각적으로 보기 어려움 (심각도: 낮음)

**Unexpected Insight**:
프로젝트 관리 도구보다 "AI가 한 일 리뷰" 기능에 더 관심을 보임.
-> 제품 포지셔닝에서 "AI 작업 리뷰" 강조 검토 필요.

**Product Fit Assessment**:
Goal-Task 계층 구조가 이 페르소나의 "오늘 할 일" 관리에 적합.
AI 실행 엔진이 완성되면 핵심 가치 제공 가능.
```

#### 실전 팁
> 💡 페르소나를 만들 때 인구통계(나이, 직업)보다 **행동 패턴과 동기**에 집중하라. "28세 개발자"보다 "퇴근 후 2시간 집중 코딩하는 사람"이 제품 의사결정에 훨씬 유용하다.

#### SprintX에서의 활용
PO 가이드 Phase 1에서 Beachhead 세그먼트를 선택한 후, 해당 세그먼트 내에서 구체적인 페르소나를 만들 때 사용한다. PRD에 "1인 개발자"라고만 적혀 있는 상태에서, 이 스킬로 "어떤 종류의 1인 개발자인지"를 구체화할 수 있다. Week 1 사용자 인터뷰 결과를 입력 데이터로 활용하면 가장 효과적이다.

#### 관련 스킬
- **이전**: `market-segments` -- 세그먼트를 먼저 식별한 후, 각 세그먼트의 페르소나를 만든다
- **다음**: `customer-journey-map` -- 페르소나가 제품을 경험하는 전체 여정을 매핑한다
- **보완**: `user-segmentation` -- 실제 사용자 데이터가 있다면 행동 기반 세분화로 페르소나를 검증한다

### 2. 시장 세그먼트 (market-segments)
**난이도**: 초급
**예상 소요 시간**: 15-20분
**슬래시 명령어**: `/pm-market-research:market-segments`

#### 이 스킬은 언제 쓰나요?
- 제품의 잠재 고객 그룹을 체계적으로 식별하고 싶을 때
- "어떤 고객을 먼저 공략할까" 결정을 위한 판단 근거가 필요할 때
- 새로운 시장 진출이나 피벗을 검토할 때

#### 실행 방법
```
/pm-market-research:market-segments SprintX - AI 에이전트 기반 스프린트 관리 도구.
현재 타겟: 1인 개발자.
기능: Goal-Task 계층 관리, AI 에이전트 실행, 코드 리뷰 플로우.
가격: Free/Pro($5/월).
```

시장 조사 보고서, 고객 데이터베이스, 기존 세그먼트 문서가 있다면 함께 제공한다.

#### 산출물 예시
```
## 세그먼트 1: AI-First 사이드 프로젝트 개발자

**규모**: 전체 시장의 약 15% (성장세)
**성장 궤도**: AI 코딩 도구 채택률 증가와 함께 빠르게 성장

**Key Demographics**:
- 25-35세, 소프트웨어 엔지니어
- Claude Code, GitHub Copilot, Cursor 등 AI 코딩 도구 활발히 사용
- 주로 개인 프로젝트 1-3개 동시 진행

**Jobs-to-be-Done**:
AI가 생성한 코드를 효율적으로 관리하고 검토하고 싶다.
주 3-5회, 프로젝트당 주 5-10시간 투자.

**Product Fit Analysis**:
SprintX의 "Goal -> Task -> AI 실행 -> 리뷰" 플로우가
이 세그먼트의 핵심 워크플로우와 정확히 일치함.
Free 플랜으로 진입, 프로젝트 수 증가 시 Pro 전환 기대.
```

#### 실전 팁
> 💡 세그먼트는 **측정 가능하고(Measurable), 접근 가능하고(Accessible), 구별 가능한(Distinct)** 것이어야 한다. "개발자"는 너무 넓고, "2026년 3월에 Claude Code를 쓴 한국인 개발자"는 너무 좁다. 적절한 크기를 찾아라.

#### SprintX에서의 활용
PO 가이드 Phase 1 Step 1-2에서 핵심적으로 사용한다. PRD의 "1인 개발자"에서 출발하여 3-5개 세그먼트로 세분화하고, 각 세그먼트의 크기/접근성/지불 의향을 비교한다. 이 스킬의 산출물은 바로 다음 단계인 `beachhead-segment`(Go-to-Market 카테고리)의 직접적인 입력이 된다.

#### 관련 스킬
- **이전**: 없음 (시장 조사의 출발점)
- **다음**: `beachhead-segment` (Go-to-Market) -- 식별한 세그먼트 중 1차 타겟을 선택한다
- **보완**: `user-personas` -- 선택한 세그먼트의 구체적인 사용자 프로필을 만든다

### 3. 사용자 세분화 (user-segmentation)
**난이도**: 중급
**예상 소요 시간**: 25-35분
**슬래시 명령어**: `/pm-market-research:user-segmentation`

#### 이 스킬은 언제 쓰나요?
- 실제 사용자 피드백 데이터(설문, 지원 티켓, 사용 로그)가 쌓여 있을 때
- 인구통계가 아닌 **행동과 동기** 기반으로 사용자 그룹을 나누고 싶을 때
- 어떤 사용자 그룹에 투자하고, 어떤 그룹을 우선순위에서 내릴지 결정해야 할 때

#### 실행 방법
```
/pm-market-research:user-segmentation SprintX 사용자 피드백 분석.
데이터: 설문 응답 50건, 지원 티켓 30건, 제품 사용 로그 요약.
목표: 행동 기반 세그먼트 식별 + 각 세그먼트의 제품 적합도 평가.
```

CSV, 설문 응답, 인터뷰 기록, 지원 티켓, 사용 로그 파일을 함께 제공하면 데이터 기반 세분화가 가능하다.

#### 산출물 예시
```
## 세그먼트 A: "체계적 관리자" (전체의 약 40%)

**행동 특성**:
- SprintX를 매일 사용, 평균 세션 15분
- Goal -> Sprint -> Task 계층을 적극 활용
- Task를 세분화하여 AI에게 위임하는 패턴

**JTBD & 동기**:
핵심 과업: 여러 프로젝트의 진행 상황을 한눈에 파악하고 싶다
동기: 통제감, 예측 가능성
성공 기준: "이번 주에 뭘 했고 다음 주에 뭘 할지 5분 안에 답할 수 있다"

**차별화된 가치 제안**:
"프로젝트 대시보드 + 진행률 시각화"를 강조하면 이 세그먼트의
전환율과 리텐션이 가장 높을 것으로 예상.

**세그먼트 우선순위**:
전략적 중요도: 높음 (성장 잠재력, 수익 영향, 비전 정렬)
구현 난이도: 중간 (기존 기능으로 부분 충족)
권고: **투자** -- 핵심 세그먼트로 집중
```

#### 실전 팁
> 💡 `market-segments`가 "시장에 어떤 고객이 있는가"라면, `user-segmentation`은 "우리 제품을 이미 쓰는/시도한 사람들 중 어떤 패턴이 있는가"이다. 전자는 출시 전, 후자는 출시 후에 더 유용하다.

#### SprintX에서의 활용
Phase 3(격주 반복) 단계에서 사용자 데이터가 쌓이면 실행한다. 초기에는 `market-segments`로 시장 가설을 세우고, 사용자가 생긴 후에는 `user-segmentation`으로 실제 행동 기반 세분화를 수행한다. Supabase의 사용자 활동 데이터와 온보딩 퍼널 SQL 결과를 입력으로 활용할 수 있다.

#### 관련 스킬
- **이전**: `market-segments` -- 시장 가설 기반 세그먼트 (출시 전)
- **다음**: `sentiment-analysis` -- 각 세그먼트의 만족도를 정량적으로 측정
- **보완**: `user-personas` -- 세분화된 그룹의 대표 페르소나 생성
- **보완**: `cohort-analysis` (Data Analytics) -- 세그먼트별 리텐션 추적

### 4. 고객 여정 맵 (customer-journey-map)
**난이도**: 초급
**예상 소요 시간**: 20-30분
**슬래시 명령어**: `/pm-market-research:customer-journey-map`

#### 이 스킬은 언제 쓰나요?
- 사용자가 제품을 발견하고, 가입하고, 사용하고, 추천하는 전체 경험을 시각화하고 싶을 때
- 사용자가 어디서 이탈하는지(drop-off point)를 찾아 개선하고 싶을 때
- 온보딩 경험을 개선하거나 활성화(activation) 지표를 정의하려 할 때

#### 실행 방법
```
/pm-market-research:customer-journey-map SprintX 사용자 여정 분석.
페르소나: AI 코딩 에이전트를 쓰는 1인 개발자.
기존 퍼널 데이터: landing_visited -> login_started -> onboarding_completed
-> dashboard_entered -> project_created -> first_sprint_created
-> first_goal_created -> first_task_created
```

인터뷰 기록, 설문 데이터, 분석 로그, 지원 티켓, 기존 여정 맵 파일이 있다면 함께 제공한다.

#### 산출물 예시
```
## 고객 여정 맵: AI-First 개발자

### 여정 단계 요약

| 단계 | 터치포인트 | 사용자 행동 | 감정 | 고통점 | 기회 |
|------|-----------|------------|------|--------|------|
| 인지 | 개발 커뮤니티, X/Twitter | "AI 프로젝트 관리 도구?" 검색 | 호기심 | 비슷한 도구가 많아 구별 어려움 | 차별화된 한 줄 메시지 |
| 가입 | 랜딩 페이지, 회원가입 | 무료 가입 시도 | 기대 | 가격 정보 불일치로 혼란 | 가격 투명성 |
| 온보딩 | 온보딩 위자드 | 첫 프로젝트 생성 | 집중 | "다음에 뭘 해야 하지?" | 가이드 투어 |
| 활성화 | 대시보드 | 첫 Task 생성 | 성취감/불안 | AI 실행이 없어 수동 작업 | Coming Soon 안내 |
| 유지 | 주간 대시보드 | 진행 상황 확인 | 만족/권태 | 업데이트 알림 부재 | 주간 요약 이메일 |

### 핵심 순간 (Critical Moments)
- **Aha Moment**: 첫 Goal-Task 계층 생성 후 "이렇게 정리되는구나"
- **Moment of Truth**: AI 실행 기능 시도 시점 (현재 미구현)
- **Churn Trigger**: Task 생성 후 AI 실행 없이 수동 완료만 반복
```

#### 실전 팁
> 💡 여정 맵의 핵심은 **감정 곡선**이다. 사용자가 기대감이 높은 순간에 실망을 주면 이탈이 가속된다. 특히 "AI가 실행해줍니다"라고 안내했는데 실제로는 없는 경우가 대표적이다. 정직한 메시지가 여정 전체의 신뢰를 유지한다.

#### SprintX에서의 활용
PO 가이드 Phase 1 Step 1-6에서 핵심적으로 사용한다. SprintX에는 이미 8단계 온보딩 퍼널 SQL(`scripts/onboarding-funnel.sql`)이 있다. 이 퍼널의 각 단계를 여정 맵의 골격으로 활용하고, 각 단계에서의 감정/고통/기회를 매핑한다. Analytics 코드의 13개 이벤트와 연동하면, 여정의 각 단계를 정량적으로 측정할 수 있다.

#### 관련 스킬
- **이전**: `user-personas` -- 여정 맵의 주인공(페르소나)을 먼저 정의한다
- **다음**: `sql-queries` (Data Analytics) -- 여정 각 단계의 전환율을 쿼리로 측정한다
- **보완**: `north-star-metric` (Marketing & Growth) -- 여정에서 가장 중요한 순간을 핵심 지표로 선정한다

### 5. 시장 규모 추정 (market-sizing)
**난이도**: 중급
**예상 소요 시간**: 30-40분
**슬래시 명령어**: `/pm-market-research:market-sizing`

#### 이 스킬은 언제 쓰나요?
- 시장 기회의 크기를 숫자로 추정하고 싶을 때 (투자 유치, 사업 계획)
- TAM/SAM/SOM을 체계적으로 계산해야 할 때
- 새로운 시장 진출이나 피벗의 경제적 타당성을 검토할 때

#### 실행 방법
```
/pm-market-research:market-sizing SprintX - AI 에이전트 기반 스프린트 관리 SaaS.
시장: 글로벌 프로젝트 관리 소프트웨어 시장 중 AI 코딩 에이전트 사용 개발자.
가격: Free / Pro $5/월.
지역: 글로벌 (한국어/영어 지원).
```

산업 보고서, 시장 조사 자료, 경쟁사 재무 데이터, 고객 정보를 함께 제공하면 더 정확한 추정이 가능하다. 웹 검색으로 최신 시장 데이터를 자동 수집한다.

#### 산출물 예시
```
## SprintX 시장 규모 추정

| 지표 | 현재 추정 | 2-3년 후 전망 |
|------|----------|-------------|
| TAM | $12.5B (글로벌 PM 소프트웨어) | $18B |
| SAM | $850M (AI 도구 사용 개발자 대상) | $2.1B |
| SOM | $4.2M (첫 3년 목표) | $12M |

### Top-down 추정:
글로벌 프로젝트 관리 소프트웨어 시장 $12.5B (2026) x
AI 코딩 도구 사용자 비율 6.8% = SAM $850M

### Bottom-up 추정:
AI 코딩 에이전트 사용 개발자 약 300만 명 x
PM 도구 유료 전환율 15% x Pro $60/년 = SAM $270M
(보수적 추정. 팀 확장 시 상향)

### 핵심 가정 (검증 필요):
1. AI 코딩 에이전트 사용 개발자 수 300만 명 (신뢰도: 중간)
2. PM 도구 유료 전환율 15% (신뢰도: 낮음 -- 검증 필요)
3. Pro 플랜 가격 $5/월 유지 (신뢰도: 높음)
```

#### 실전 팁
> 💡 TAM/SAM/SOM 추정에서 가장 중요한 것은 **가정을 명시하는 것**이다. 숫자 자체보다 "이 숫자가 맞으려면 어떤 조건이 필요한가"를 파악하는 것이 의사결정에 더 유용하다. Top-down과 Bottom-up 두 가지를 모두 계산해서 교차 검증하라.

#### SprintX에서의 활용
Phase 1에서 Beachhead 세그먼트를 선택한 후, 해당 세그먼트의 시장 크기를 추정할 때 사용한다. 투자 유치나 사업 계획서 작성에도 활용된다. 경쟁 분석(Step 1-3)의 경쟁자 매출/고객 수 데이터를 Bottom-up 추정의 참조점으로 사용할 수 있다.

#### 관련 스킬
- **이전**: `market-segments` -- 세그먼트를 식별한 후 각 세그먼트의 규모를 추정한다
- **이전**: `competitor-analysis` -- 경쟁자 데이터가 시장 규모 교차 검증에 활용된다
- **다음**: `beachhead-segment` (Go-to-Market) -- 시장 규모를 기준으로 Beachhead를 최종 선택한다
- **보완**: `gtm-strategy` (Go-to-Market) -- SOM 달성을 위한 구체적 전략 수립

### 6. 경쟁 분석 (competitor-analysis)
**난이도**: 초급
**예상 소요 시간**: 20-30분
**슬래시 명령어**: `/pm-market-research:competitor-analysis`

#### 이 스킬은 언제 쓰나요?
- 경쟁 환경을 체계적으로 파악하고 싶을 때
- "왜 SprintX를 선택해야 하는가"에 대한 명확한 답이 필요할 때
- 차별화 포인트를 발견하고 포지셔닝 전략을 수립하려 할 때

#### 실행 방법
```
/pm-market-research:competitor-analysis SprintX - AI 에이전트 기반 스프린트 관리 SaaS.
시장: AI 코딩 에이전트 관리 + 프로젝트 관리.
알려진 경쟁자: Linear, Jira, GitHub Projects.
AI 에이전트 관리: Devin, SWE-Agent, Sweep.
기존 분석: SYMPHONY-vs-SPRINTX-GAP.md 참조.
```

시장 조사 자료, 경쟁사 데이터, 가격 시트, 기능 비교표, 고객 피드백을 함께 제공하면 더 깊은 분석이 가능하다. 웹 검색으로 경쟁자 정보를 자동 수집한다.

#### 산출물 예시
```
## 경쟁자 프로필: Linear

**회사**: Linear, 2019년 설립, $52M 펀딩
**포지셔닝**: "소프트웨어 팀을 위한 이슈 트래커"
**타겟**: 스타트업/중소 개발팀

**핵심 강점**:
- 빠른 UI/UX (업계 최고 수준)
- 키보드 단축키 중심 워크플로우
- 강력한 GitHub/GitLab 통합

**핵심 약점**:
- AI 에이전트 실행/관리 기능 없음
- 1인 개발자에게는 과도한 팀 기능
- 무료 플랜 제한적

**SprintX 차별화 기회**:
Linear가 "팀을 위한 이슈 트래커"라면,
SprintX는 "AI와 일하는 개발자를 위한 프로젝트 관리"로 포지셔닝.
AI 에이전트 실행 + 리뷰 플로우가 핵심 차별점.
```

#### 실전 팁
> 💡 경쟁 분석에서 가장 흔한 실수는 "기능 비교표만 만드는 것"이다. 중요한 것은 **고객이 왜 그 제품을 선택하는가**(또는 떠나는가)이다. G2, Capterra의 리뷰, Reddit의 불만 글에서 더 깊은 인사이트를 얻을 수 있다.

#### SprintX에서의 활용
PO 가이드 Phase 1 Step 1-3에서 핵심적으로 사용한다. 기존에 작성된 `SYMPHONY-vs-SPRINTX-GAP.md`를 seed 데이터로 활용하여 cold-start를 방지한다. 이 분석의 결과는 가치 제안(Value Proposition) 확정과 가격 전략 수립의 직접적인 입력이 된다. 경쟁자의 가격을 벤치마크하면 "$5/월이 적절한가"를 판단할 수 있다.

#### 관련 스킬
- **이전**: `market-segments` -- 어떤 시장에서의 경쟁인지 먼저 정의한다
- **다음**: `competitive-battlecard` (Go-to-Market) -- 경쟁 분석을 영업/마케팅용 배틀카드로 전환한다
- **보완**: `positioning-ideas` (Marketing & Growth) -- 경쟁 분석의 차별화 포인트를 포지셔닝 전략으로 발전시킨다

### 7. 감성 분석 (sentiment-analysis)
**난이도**: 고급
**예상 소요 시간**: 35-45분
**슬래시 명령어**: `/pm-market-research:sentiment-analysis`

#### 이 스킬은 언제 쓰나요?
- 사용자 피드백(리뷰, 설문, 지원 티켓)이 많이 쌓여서 수작업으로 분석하기 어려울 때
- 세그먼트별로 만족도 차이를 정량적으로 비교하고 싶을 때
- 제품 개선의 우선순위를 "가장 불만족한 영역"부터 매기고 싶을 때

#### 실행 방법
```
/pm-market-research:sentiment-analysis SprintX 사용자 피드백 감성 분석.
데이터: 사용자 리뷰 100건 (CSV 첨부),
Waitlist 피드백 30건, 지원 티켓 50건.
목표: 세그먼트별 만족도 점수와 개선 우선순위 도출.
```

CSV, PDF, 설문 응답, 리뷰 데이터, 소셜 리스닝 보고서 등 피드백 소스 파일을 반드시 함께 제공해야 한다.

#### 산출물 예시
```
## 세그먼트별 감성 분석 결과

### 세그먼트 1: 체계적 관리자
**감성 점수**: +0.62 (긍정)
**만족 요인**: Goal-Task 계층 구조, 직관적 UI
**불만 요인**: AI 실행 기능 미구현, 모바일 미지원
**NPS 근사**: 45 (양호)

**긍정 피드백 주요 테마**:
- "할 일을 체계적으로 정리할 수 있어서 좋다" (12회 언급)
- "칸반 보드가 깔끔하다" (8회)

**부정 피드백 주요 테마**:
- "AI가 실행해준다더니 안 되더라" (15회 -- 최우선 개선 필요)
- "알림이 없어서 확인을 잊는다" (7회)
- 원문 인용: "AI 실행이 핵심인 줄 알고 가입했는데 Coming Soon이라니..."

**우선 조치 권고**:
1. (즉시) AI 실행 메시지를 정직하게 수정 -- 기대 관리
2. (단기) 알림/이메일 기능 추가
3. (중기) AI 실행 엔진 완성
```

#### 실전 팁
> 💡 감성 분석은 **데이터 양이 충분할 때** 의미가 있다. 피드백이 20건 미만이면 이 스킬 대신 `user-segmentation`이나 직접 인터뷰가 더 효과적이다. 최소 50건 이상의 피드백이 있을 때 실행을 권장한다.

#### SprintX에서의 활용
Phase 3(격주 반복)에서 사용자 피드백이 충분히 쌓인 후 실행한다. 초기에는 Waitlist 등록 시 수집한 간단한 피드백, 지원 요청, 커뮤니티 반응 등을 모아 분석할 수 있다. 분석 결과의 세그먼트별 만족도 점수는 백로그 재정렬과 OKR 체크인의 정량적 근거가 된다.

#### 관련 스킬
- **이전**: `user-segmentation` -- 세그먼트를 먼저 정의한 후, 각 세그먼트의 감성을 분석한다
- **다음**: `cohort-analysis` (Data Analytics) -- 감성 점수와 리텐션의 상관관계를 분석한다
- **보완**: `competitor-analysis` -- 경쟁자 리뷰의 감성 분석으로 비교 우위를 파악한다
