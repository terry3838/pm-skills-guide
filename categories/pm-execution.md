# 실행 및 관리 (Execution)

## 카테고리 소개

전략을 세우는 것만으로는 제품이 만들어지지 않습니다. **실행 및 관리**는 전략 수립 이후 실제로 제품을 개발하고, 출시하고, 개선하는 핵심 단계를 다룹니다.

비개발자 PO(Product Owner)에게 이 카테고리가 특히 중요한 이유는, AI 코딩 에이전트에게 작업을 지시할 때 **무엇을 만들어야 하는지(PRD)**, **어떤 순서로 만들어야 하는지(우선순위/스프린트)**, **어떻게 검증해야 하는지(테스트 시나리오)** 를 명확하게 전달해야 하기 때문입니다. 이 카테고리의 15개 스킬은 "생각을 구조화된 문서로 바꾸는 도구"로서, 전략과 개발 사이의 다리 역할을 합니다.

SprintX PO 가이드에서는 Phase 2(실행 계획 + 데이터 수집)와 Phase 3(격주 반복 루프)에서 이 카테고리의 스킬을 집중적으로 사용합니다.

## 이 카테고리의 스킬 한눈에 보기

| 스킬명 | 한 줄 설명 | 난이도 | 예상 소요 시간 |
|--------|-----------|--------|--------------|
| PRD 작성 (create-prd) | 8섹션 템플릿으로 제품 요구사항 문서 작성 | 초급 | 30~60분 |
| OKR 브레인스토밍 (brainstorm-okrs) | 회사 전략에 정렬된 팀 OKR 3세트 생성 | 초급 | 20~30분 |
| 결과 기반 로드맵 (outcome-roadmap) | 기능 나열형 로드맵을 결과 중심으로 전환 | 중급 | 30~45분 |
| 스프린트 계획 (sprint-plan) | 팀 역량 추정, 스토리 선택, 의존성 매핑 | 중급 | 30~45분 |
| 스프린트 회고 (retro) | 구조화된 회고 진행 및 액션 아이템 도출 | 초급 | 20~30분 |
| 릴리스 노트 (release-notes) | 기술 티켓을 사용자 친화적 릴리스 노트로 변환 | 초급 | 15~20분 |
| 사전 위험 분석 (pre-mortem) | 출시 전 실패 시나리오 분석 및 대응 계획 수립 | 중급 | 30~45분 |
| 이해관계자 맵 (stakeholder-map) | Power x Interest 그리드로 이해관계자 분류 및 커뮤니케이션 계획 | 중급 | 20~30분 |
| 회의록 요약 (summarize-meeting) | 회의 녹취/메모를 구조화된 요약으로 변환 | 초급 | 10~15분 |
| 사용자 스토리 (user-stories) | 3C + INVEST 기준으로 사용자 스토리 작성 | 초급 | 20~30분 |
| Job 스토리 (job-stories) | JTBD 기반 "When-I want-So I can" 형식 스토리 작성 | 중급 | 20~30분 |
| WWA 백로그 (wwas) | Why-What-Acceptance 형식의 백로그 아이템 작성 | 중급 | 20~30분 |
| 테스트 시나리오 (test-scenarios) | 사용자 스토리에서 QA 테스트 시나리오 자동 생성 | 중급 | 20~30분 |
| 더미 데이터셋 (dummy-dataset) | 테스트용 현실적 더미 데이터 생성 (CSV/JSON/SQL/Python) | 초급 | 10~20분 |
| 우선순위 프레임워크 (prioritization-frameworks) | RICE, ICE 등 9개 프레임워크 가이드 및 적용 | 중급 | 30~45분 |

## 추천 실행 순서

이 카테고리의 스킬은 제품 개발 생명주기를 따라 자연스럽게 흐릅니다. 아래 순서는 SprintX PO 가이드의 Phase 2~3에 맞춰 설계되었습니다.

### 1단계: 계획 수립 (Phase 2 초반)
1. **prioritization-frameworks** -- 백로그 25개 이슈를 RICE/ICE로 재정렬
2. **brainstorm-okrs** -- Q2 2026 OKR 설정으로 방향 확인
3. **pre-mortem** -- 실패 시나리오 3~5개를 미리 분석
4. **outcome-roadmap** -- 기능 목록을 결과 기반 로드맵으로 전환

### 2단계: 개발 핸드오프 (Phase 2 후반)
5. **create-prd** -- 최우선 기능 1~2개에 대해 PRD 작성
6. **user-stories** 또는 **job-stories** 또는 **wwas** -- PRD를 개발 가능한 단위로 분해
7. **test-scenarios** -- 사용자 스토리에서 QA 시나리오 파생
8. **dummy-dataset** -- 프로토타이핑/테스트용 데이터 생성

### 3단계: 스프린트 실행 (Phase 3 반복)
9. **sprint-plan** -- 격주 스프린트 계획
10. **retro** -- 격주 스프린트 회고

### 4단계: 출시 및 소통 (수시)
11. **release-notes** -- 기능 배포 후 사용자 공지
12. **summarize-meeting** -- 사용자 인터뷰/회의 정리
13. **stakeholder-map** -- 이해관계자 관리 (필요 시)

## 관련 카테고리

- **전략 (Product Strategy)**: 실행의 전제가 되는 전략 캔버스, 가치 제안, 가격 전략 등. 실행 전에 전략 카테고리의 스킬을 먼저 사용합니다.
- **시장 조사 (Market Research)**: 경쟁 분석, 시장 세그먼트, 고객 여정 맵 등. 우선순위 결정과 PRD 작성의 입력이 됩니다.
- **제품 발견 (Product Discovery)**: 실험 브레인스토밍 등. pre-mortem과 outcome-roadmap의 근거가 됩니다.
- **GTM (Go-to-Market)**: GTM 전략, Beachhead 세그먼트 등. release-notes와 stakeholder-map이 GTM 실행을 지원합니다.
- **PM 도구함 (PM Toolkit)**: 문법 체크, 개인정보 처리방침 등. 실행 산출물의 품질 검수에 활용합니다.

## 스킬 상세

### 1. PRD 작성 (create-prd)
**난이도**: 초급
**예상 소요 시간**: 30~60분
**슬래시 명령어**: `/pm-execution:create-prd`

#### 이 스킬은 언제 쓰나요?
- 새로운 기능을 개발하기 전에 요구사항을 정리할 때
- AI 에이전트에게 개발을 지시하기 전에 "무엇을 왜 만드는지" 명확히 해야 할 때
- 기존 PRD를 리뷰하거나 보완할 때

#### 실행 방법
```
/pm-execution:create-prd SprintX AI 실행 대시보드
```
AI가 8개 섹션에 대해 질문하고, 답변을 바탕으로 PRD를 생성합니다.

#### 산출물 예시
8개 섹션으로 구성된 마크다운 PRD 문서:
```
1. Summary (2~3문장 요약)
2. Contacts (핵심 이해관계자)
3. Background (배경과 "왜 지금인가")
4. Objective (목표 + SMART OKR 형식 Key Results)
5. Market Segment(s) (대상 사용자와 제약 조건)
6. Value Proposition(s) (고객 과업, 이득, 고통, 경쟁 우위)
7. Solution (UX/프로토타입, 핵심 기능, 기술, 가정)
8. Release (단계별 출시 계획)
```

#### 실전 팁
> 💡 "이것을 만들어줘"보다 "이런 사용자가 이런 문제를 겪고 있고, 이것이 해결되면 이런 지표가 움직인다"라고 전달하면 AI 에이전트가 훨씬 좋은 결과를 만듭니다. PRD는 그 전달 문서입니다.

#### SprintX에서의 활용
PO 가이드 **Phase 2 Step 2-5**에서 사용합니다. Step 2-1에서 확정된 최우선 기능 1~2개에 대해 PRD를 작성합니다. 이 PRD가 AI 에이전트에게 전달할 개발 지시서가 됩니다. 문제 정의, 사용자 시나리오, 성공 지표, 범위(In scope / Out of scope)를 명확히 포함하세요.

#### 관련 스킬
- **이전**: prioritization-frameworks (우선순위 확정 후 PRD 작성)
- **다음**: user-stories, job-stories, wwas (PRD를 개발 단위로 분해)
- **보완**: pre-mortem (PRD 기반 위험 분석)

### 2. OKR 브레인스토밍 (brainstorm-okrs)
**난이도**: 초급
**예상 소요 시간**: 20~30분
**슬래시 명령어**: `/pm-execution:brainstorm-okrs`

#### 이 스킬은 언제 쓰나요?
- 분기 OKR을 설정할 때
- 팀 목표를 회사 전략과 정렬시키고 싶을 때
- 효과적인 OKR 작성법을 배우고 싶을 때

#### 실행 방법
```
/pm-execution:brainstorm-okrs SprintX Q2 2026 제품팀
```
회사 전략, 팀 맥락, 현재 상황을 입력하면 3가지 대안적 OKR 세트를 생성합니다.

#### 산출물 예시
각 세트는 아래 형식을 따릅니다:
```
Objective: 신규 사용자에게 놀라운 온보딩 경험을 제공한다
Key Results:
- 온보딩 완료율 66% 이상 달성
- 평균 가치 체험 시간(TTV) 20분 이내
- 온보딩 설문 CSAT 75% 이상
```
OKR, KPI, North Star Metric의 관계도 함께 설명됩니다.

#### 실전 팁
> 💡 Key Results는 "기능 5개 출시" 같은 산출물(output)이 아니라 "온보딩 완료율 66% 이상" 같은 결과(outcome) 지표로 작성하세요. 3가지 세트 중 하나를 고르는 것이 아니라, 세 가지를 보며 전략적 토론을 촉발하는 것이 목적입니다.

#### SprintX에서의 활용
PO 가이드 **Phase 2 Step 2-2**에서 사용합니다. "첫 10명 외부 사용자가 SprintX로 프로젝트를 관리한다" 같은 Objective를 설정하고, 측정 가능한 Key Results(예: "Beachhead 세그먼트에서 5명 인터뷰 완료")를 정의합니다.

#### 관련 스킬
- **이전**: prioritization-frameworks (백로그 정렬 후 OKR 설정)
- **다음**: outcome-roadmap (OKR에 정렬된 로드맵 작성)
- **보완**: North Star Metric 정의 (pm-marketing-growth 카테고리)

### 3. 결과 기반 로드맵 변환 (outcome-roadmap)
**난이도**: 중급
**예상 소요 시간**: 30~45분
**슬래시 명령어**: `/pm-execution:outcome-roadmap`

#### 이 스킬은 언제 쓰나요?
- 기능 나열형 로드맵("Q2: 검색 필터 구현, AI 추천 구현")을 전략적 로드맵으로 전환할 때
- 로드맵을 이해관계자에게 발표하면서 "왜 이것을 만드는가"를 설명해야 할 때
- 로드맵을 더 유연하게 만들어 방향 전환에 대비하고 싶을 때

#### 실행 방법
```
/pm-execution:outcome-roadmap SprintX 2026 Q2-Q3 로드맵
```
현재 로드맵(기능 목록)을 입력하면, 각 항목을 결과 중심 문장으로 변환합니다.

#### 산출물 예시
**변환 전 (Output 중심)**:
```
Q2: 고급 검색 필터 구현, AI 추천 구현, 대시보드 리디자인
```
**변환 후 (Outcome 중심)**:
```
Q2: 사용자가 원하는 제품을 50% 더 빠르게 찾을 수 있도록 직관적 탐색 경험 제공
Q2: 개인화된 AI 추천을 통해 평균 주문 금액 20% 증가
Q2: 대시보드 로딩 시간 80% 단축으로 운영자가 모든 시스템을 한눈에 모니터링
```

#### 실전 팁
> 💡 기능 하나가 어떤 결과를 만드는지 모르겠다면 "그래서 뭐?" 질문을 반복하세요. "검색 필터 추가" → "그래서 뭐?" → "사용자가 원하는 걸 더 빨리 찾음" → "그래서 뭐?" → "이탈률 감소". 마지막 답이 결과(outcome)입니다.

#### SprintX에서의 활용
PO 가이드 **Phase 2 Step 2-4**에서 pre-mortem 직후 사용합니다. 25개 오픈 이슈(#161~#185)를 기능 목록이 아닌 결과 기반으로 재구성합니다. 3개월 단위 월별 목표 결과, 결과 달성에 필요한 기능/실험 매핑, 의존성과 리스크를 포함합니다.

#### 관련 스킬
- **이전**: brainstorm-okrs (OKR이 로드맵 방향을 결정), pre-mortem (리스크가 로드맵 순서에 영향)
- **다음**: create-prd (로드맵 최우선 항목의 PRD 작성)
- **보완**: sprint-plan (로드맵의 분기 목표를 스프린트 단위로 분해)

### 4. 스프린트 계획 (sprint-plan)
**난이도**: 중급
**예상 소요 시간**: 30~45분
**슬래시 명령어**: `/pm-execution:sprint-plan`

#### 이 스킬은 언제 쓰나요?
- 다음 스프린트에 어떤 작업을 넣을지 계획할 때
- 팀 역량 대비 적절한 작업량을 산정할 때
- 작업 간 의존성과 리스크를 파악해야 할 때

#### 실행 방법
```
/pm-execution:sprint-plan SprintX Sprint 5 (2026-03-18 ~ 2026-03-31)
```
백로그, 과거 속도(velocity) 데이터, 팀 구성을 입력하면 스프린트 계획을 생성합니다.

#### 산출물 예시
```
Sprint Goal: Beachhead 세그먼트 사용자가 프로젝트를 생성하고 첫 Goal을 설정할 수 있다
Duration: 2주
Team Capacity: 40 story points
Committed Stories: 35 story points (8개 스토리)
Buffer: 5 points (12.5%)

Stories:
1. 온보딩 플로우 개선 -- 8pt -- PO -- 없음
2. Goal 생성 UX 개선 -- 5pt -- AI 에이전트 -- 스토리 1 완료 후
...

Risks:
- AI 에이전트 응답 지연 시 스토리 2~4 병목 → 비동기 작업 구조로 대응
```

#### 실전 팁
> 💡 역량의 15~20%는 예상치 못한 작업(버그, 기술 부채)을 위해 비워두세요. 매 스프린트 "할 수 있는 만큼"이 아니라 "확실히 끝낼 수 있는 만큼"을 넣는 것이 핵심입니다.

#### SprintX에서의 활용
PO 가이드 **Phase 3**에서 격주 스프린트 계획으로 사용합니다. 1인 PO + AI 에이전트 구조에서는 팀 역량 추정을 "PO가 리뷰할 수 있는 작업량" 기준으로 조정합니다. Definition of Ready(명확한 인수 조건, 추정 완료, 블로커 없음)를 확인하세요.

#### 관련 스킬
- **이전**: create-prd (PRD에서 스프린트 스토리 파생), prioritization-frameworks (우선순위 기반 스토리 선택)
- **다음**: retro (스프린트 종료 후 회고)
- **보완**: user-stories (스프린트에 넣을 스토리 형식)

### 5. 스프린트 회고 (retro)
**난이도**: 초급
**예상 소요 시간**: 20~30분
**슬래시 명령어**: `/pm-execution:retro`

#### 이 스킬은 언제 쓰나요?
- 스프린트가 끝나고 무엇이 잘되었고 무엇을 개선할지 정리할 때
- 팀 피드백(설문, 슬랙 메시지 등)을 체계적으로 분석할 때
- 이전 회고의 액션 아이템 이행 여부를 점검할 때

#### 실행 방법
```
/pm-execution:retro SprintX Sprint 4 회고
```
스프린트 데이터(속도, 목표 달성 여부), 팀 피드백을 입력하면 구조화된 회고 문서를 생성합니다.

#### 산출물 예시
3가지 회고 형식 중 선택 가능:
- **Start / Stop / Continue**: 시작할 것, 멈출 것, 계속할 것
- **4Ls** (Liked / Learned / Lacked / Longed For): 좋았던 것, 배운 것, 부족했던 것, 원했던 것
- **Sailboat**: 바람(추진력), 닻(장애물), 암초(위험), 섬(목표)

```
## Sprint 4 Retrospective -- 2026-03-14

### Sprint Performance
- Goal: 부분 달성 (3/5 스토리 완료)
- Committed: 35 pts | Completed: 25 pts

### Action Items
1. 스토리 추정 정확도 개선 -- PO -- 3/21까지
2. AI 에이전트 작업 리뷰 프로세스 정립 -- PO -- 3/21까지
```

#### 실전 팁
> 💡 액션 아이템은 2~3개로 제한하세요. 더 많으면 하나도 실행되지 않습니다. 각 액션 아이템에는 반드시 담당자, 기한, 성공 기준을 포함하세요.

#### SprintX에서의 활용
PO 가이드 **Phase 3**에서 격주 회고로 사용합니다. 1인 PO 구조에서는 "나 자신의 작업 패턴 회고"로 활용합니다. 이전 회고 액션 아이템의 이행 상태(완료/진행 중/미시작)를 반드시 확인합니다.

#### 관련 스킬
- **이전**: sprint-plan (스프린트 목표 대비 결과 비교)
- **다음**: sprint-plan (다음 스프린트 계획에 회고 인사이트 반영)
- **보완**: brainstorm-okrs (분기 OKR 체크인과 연계)

### 6. 릴리스 노트 작성 (release-notes)
**난이도**: 초급
**예상 소요 시간**: 15~20분
**슬래시 명령어**: `/pm-execution:release-notes`

#### 이 스킬은 언제 쓰나요?
- 기능을 배포한 후 사용자에게 변경 사항을 알릴 때
- 기술 티켓이나 Git 로그를 사용자 친화적 언어로 바꿔야 할 때
- 제품 업데이트 공지를 작성할 때

#### 실행 방법
```
/pm-execution:release-notes SprintX v0.5.0
```
JIRA 티켓, Linear 이슈, Git 로그, PRD 등을 입력하면 카테고리별로 정리된 릴리스 노트를 생성합니다.

#### 산출물 예시
```
# SprintX -- v0.5.0 (2026-03-15)

## 새로운 기능
- **다크 모드**: 눈의 피로를 줄이는 다크 테마를 지원합니다. 설정에서 전환하세요.

## 개선 사항
- **대시보드**: 로딩 속도가 3배 빨라졌습니다.

## 버그 수정
- 일부 주문이 트래픽 급증 시 실패하던 문제를 해결했습니다.
```

변환 원칙: 기술 변경("Redis 캐싱 레이어 구현")을 사용자 혜택("대시보드가 3배 빨라졌습니다")으로 바꿉니다.

#### 실전 팁
> 💡 릴리스 노트의 첫 문장은 기술적 변경이 아니라 사용자가 얻는 혜택으로 시작하세요. "Redis 캐싱을 구현했습니다"가 아니라 "대시보드가 3배 빨라져서 기다리는 시간이 줄어듭니다"가 좋습니다.

#### SprintX에서의 활용
기능 배포 후 사용자 공지 작성에 활용합니다. B2B SaaS인 SprintX의 경우 전문적이면서도 친근한 톤을 유지합니다. Waitlist 등록자에게 보내는 업데이트 이메일의 콘텐츠로도 활용 가능합니다.

#### 관련 스킬
- **이전**: sprint-plan (스프린트에서 완료된 항목이 릴리스 노트의 입력)
- **보완**: grammar-check (pm-toolkit, 릴리스 노트 품질 검수)

### 7. 사전 위험 분석 (pre-mortem)
**난이도**: 중급
**예상 소요 시간**: 30~45분
**슬래시 명령어**: `/pm-execution:pre-mortem`

#### 이 스킬은 언제 쓰나요?
- 제품 출시 전 "무엇이 잘못될 수 있는가"를 체계적으로 분석할 때
- PRD나 출시 계획의 약점을 사전에 발견하고 싶을 때
- 이해관계자에게 리스크 대응 계획을 보여줘야 할 때

#### 실행 방법
```
/pm-execution:pre-mortem SprintX AI 실행 엔진 출시
```
PRD 또는 출시 계획을 입력하면, "14일 후 출시가 실패했다고 가정"하고 역방향으로 위험을 분석합니다.

#### 산출물 예시
위험을 3가지 유형으로 분류합니다:
```
### Tigers (실제 위험)
- Worker daemon 미완성 시 핵심 가치 제안 불가 → [출시 차단]

### Paper Tigers (과장된 우려)
- "경쟁사가 같은 기능을 먼저 출시할 수 있다" → 현재 직접 경쟁자 없음

### Elephants (논의되지 않는 걱정)
- Beachhead 세그먼트가 실제로 AI 실행을 원하는가? → 조사 필요
```
Tigers는 다시 **출시 차단(Launch-Blocking)**, **빠른 후속(Fast-Follow)**, **모니터링(Track)** 으로 세분화됩니다.

#### 실전 팁
> 💡 "확신이 없으면 Tiger로 분류하세요." 위험을 과소평가하는 것보다 과대평가하는 것이 안전합니다. 출시 2~3주 전에 다시 점검하여 대응 조치가 진행 중인지 확인하세요.

#### SprintX에서의 활용
PO 가이드 **Phase 2 Step 2-4**에서 사용합니다. 실패 시나리오 3~5개를 분석합니다. SprintX의 경우 "Worker daemon(#168) 없이 AI 실행을 약속하면?" "Beachhead 세그먼트가 예상과 다르면?" 같은 시나리오가 핵심입니다.

#### 관련 스킬
- **이전**: create-prd (PRD를 기반으로 위험 분석)
- **다음**: outcome-roadmap (위험을 반영한 로드맵 조정)
- **보완**: stakeholder-map (위험 대응의 담당자 지정)

### 8. 이해관계자 맵 (stakeholder-map)
**난이도**: 중급
**예상 소요 시간**: 20~30분
**슬래시 명령어**: `/pm-execution:stakeholder-map`

#### 이 스킬은 언제 쓰나요?
- 프로젝트에 관련된 모든 사람의 관심도와 영향력을 파악할 때
- 누구에게 어떤 빈도로 어떤 채널로 소통해야 하는지 계획할 때
- 이해관계자 간 이해 충돌을 사전에 식별할 때

#### 실행 방법
```
/pm-execution:stakeholder-map SprintX AI 실행 엔진 프로젝트
```
프로젝트 설명, 조직도, 팀 구성을 입력하면 Power x Interest 그리드 기반 이해관계자 맵을 생성합니다.

#### 산출물 예시
```
| 이해관계자 | 역할 | 영향력 | 관심도 | 전략 | 빈도 | 채널 | 핵심 메시지 |
|-----------|------|--------|--------|------|------|------|-----------|
| PO | 제품 책임 | High | High | 긴밀 관리 | 매일 | Slack/대면 | 진행 상황 + 의사결정 |
| 투자자 | 자금 | High | Low | 만족 유지 | 월 1회 | 이메일 | 핵심 지표 + 마일스톤 |
```

4개 사분면별 전략:
- **High Power + High Interest**: 긴밀 관리 (정기 1:1, 의사결정 참여)
- **High Power + Low Interest**: 만족 유지 (주기적 업데이트, 중요 이슈만 에스컬레이션)
- **Low Power + High Interest**: 정보 제공 (정기 업데이트, 데모 초대)
- **Low Power + Low Interest**: 모니터링 (요청 시 정보 제공)

#### 실전 팁
> 💡 1인 PO + AI 에이전트 구조에서는 전통적 이해관계자 맵이 단순해집니다. 하지만 외부 사용자, 베타 테스터, 잠재 파트너 등을 이해관계자로 포함하면 더 유용합니다.

#### SprintX에서의 활용
1인 PO + AI 에이전트 구조에서는 간략하게 사용합니다. 외부 사용자 인터뷰 대상, Waitlist 등록자, 잠재 파트너를 이해관계자로 매핑하고, 각각에 대한 커뮤니케이션 계획을 수립합니다.

#### 관련 스킬
- **보완**: pre-mortem (위험 대응 담당자 지정에 이해관계자 맵 활용)
- **보완**: release-notes (이해관계자별 맞춤 메시지 작성)

### 9. 회의록 요약 (summarize-meeting)
**난이도**: 초급
**예상 소요 시간**: 10~15분
**슬래시 명령어**: `/pm-execution:summarize-meeting`

#### 이 스킬은 언제 쓰나요?
- 회의 녹취록이나 메모를 구조화된 요약으로 정리할 때

회의 녹취록, 녹음 파일, 메모를 입력하면 구조화된 회의 요약을 생성합니다.

```
## Meeting Summary

**Date & Time**: 2026-03-10 14:00-15:00

|----------|-------|--------|
```

#### 실전 팁
> 💡 회의 중 "이것은 결정인가, 논의인가?"를 구분하는 습관을 들이세요. 이 스킬은 그 구분을 명확히 해줍니다. 액션 아이템에는 반드시 담당자와 기한을 포함하세요.

#### SprintX에서의 활용
사용자 인터뷰 녹취 요약에 활용합니다. PO 가이드에서 Week 1부터 시작하는 사용자 인터뷰(최소 3명)의 기록을 체계적으로 정리할 수 있습니다. interview-summarize(pm-market-research 카테고리)와 유사하지만, 이 스킬은 범용 회의록에 적합합니다.

#### 관련 스킬
- **보완**: user-stories (인터뷰에서 발견된 니즈를 스토리로 전환)

### 10. 사용자 스토리 작성 (user-stories)
**난이도**: 초급
**예상 소요 시간**: 20~30분
**슬래시 명령어**: `/pm-execution:user-stories`

#### 이 스킬은 언제 쓰나요?
- 기능을 개발 가능한 작은 단위로 분해할 때
- "사용자로서, ~하고 싶다, ~하기 위해"(As a... I want... So that...) 형식으로 요구사항을 표현할 때
- 각 스토리에 명확한 인수 조건(Acceptance Criteria)을 정의할 때

#### 실행 방법
```
/pm-execution:user-stories SprintX "Goal 생성 기능" --design "Figma링크" --assumptions "1인 PO 사용 시나리오"
```
제품명, 기능, 디자인 링크, 가정을 입력하면 3C(Card, Conversation, Confirmation) + INVEST 기준의 사용자 스토리 세트를 생성합니다.

#### 산출물 예시
```
Title: 최근 본 항목 섹션

Description: 온라인 쇼핑 사용자로서, 제품 페이지에서 "최근 본 항목"을
볼 수 있기를 원한다. 이전에 고려했던 상품을 쉽게 다시 방문하기 위해.

Design: [Figma 링크]

Acceptance Criteria:
1. 이전에 1개 이상 제품을 본 사용자에게만 섹션 표시
2. 현재 보고 있는 제품은 목록에서 제외
3. 이미지, 제목, 가격이 포함된 제품 카드 표시
4. 각 카드에 "5분 전 조회" 같은 시간 표시
5. 카드 클릭 시 해당 제품 페이지로 이동
```

#### 실전 팁
> 💡 INVEST 기준을 기억하세요: Independent(독립적), Negotiable(협상 가능), Valuable(가치 있는), Estimable(추정 가능), Small(작은), Testable(테스트 가능). 하나의 스프린트 안에 완료할 수 있는 크기로 나누세요.

#### SprintX에서의 활용
PRD 이후 개발 핸드오프 시 사용합니다. AI 에이전트에게 작업을 지시할 때 사용자 스토리 형식으로 전달하면, 에이전트가 "누구를 위한 기능인지", "왜 필요한지", "완료 조건이 무엇인지" 명확히 이해합니다.

#### 관련 스킬
- **이전**: create-prd (PRD를 스토리 단위로 분해)
- **다음**: test-scenarios (스토리에서 테스트 시나리오 파생)
- **대안**: job-stories (역할 대신 상황 중심), wwas (전략적 맥락 강조)

### 11. Job 스토리 작성 (job-stories)
**난이도**: 중급
**예상 소요 시간**: 20~30분
**슬래시 명령어**: `/pm-execution:job-stories`

#### 이 스킬은 언제 쓰나요?
- 사용자 역할보다 상황과 동기에 초점을 맞추고 싶을 때
- JTBD(Jobs to Be Done) 관점에서 백로그를 작성할 때
- "When [상황], I want to [동기], so I can [결과]" 형식으로 요구사항을 표현할 때

#### 실행 방법
```
/pm-execution:job-stories SprintX "주간 스프린트 진행률 추적" --context "1인 PO가 AI 에이전트의 작업 결과를 검토하는 상황"
```
제품명, 기능, 사용자 상황을 입력하면 JTBD 기반의 Job 스토리 세트를 생성합니다.

#### 산출물 예시
```
Title: 주간 간식 지출 추적

Description: 주간 간식 용돈을 준비할 때(상황),
지금까지 얼마를 썼는지 빠르게 보고 싶다(동기),
주말 전에 돈이 바닥나지 않도록 하기 위해(결과).

Acceptance Criteria:
1. "주간 지출 현황" 섹션에 지출 요약 표시
2. 지출 기록 시 실시간 업데이트
3. 주간 예산 대비 진행률 표시 (프로그레스 바)
4. 남은 예산 금액을 눈에 띄는 색상으로 강조
5. 카테고리별 지출 상세 내역 제공
6. 예산 80% 도달 시 알림
```

#### 실전 팁
> 💡 사용자 스토리와 Job 스토리의 차이: 사용자 스토리는 "역할"(As a PO)에 초점을 맞추고, Job 스토리는 "상황"(When I'm reviewing AI agent outputs)에 초점을 맞춥니다. 같은 사용자가 다른 상황에서 다른 니즈를 가질 때 Job 스토리가 더 적합합니다.

#### SprintX에서의 활용
PRD 이후 개발 핸드오프 시 user-stories의 대안으로 사용합니다. SprintX의 JTBD인 "AI 에이전트에게 작업을 위임하고 결과를 검토"하는 다양한 상황을 포착하기에 적합합니다.

#### 관련 스킬
- **이전**: create-prd (PRD를 Job 스토리로 분해)
- **다음**: test-scenarios (Job 스토리에서 테스트 시나리오 파생)
- **대안**: user-stories (역할 중심), wwas (전략적 맥락 강조)

### 12. Why-What-Acceptance 백로그 (wwas)
**난이도**: 중급
**예상 소요 시간**: 20~30분
**슬래시 명령어**: `/pm-execution:wwas`

#### 이 스킬은 언제 쓰나요?
- 전략적 맥락(Why)을 포함한 백로그 아이템을 작성할 때
- 팀이 "왜 이것을 만드는가"를 이해하면서 개발하도록 할 때
- user-stories나 job-stories보다 간결하면서도 전략적인 형식이 필요할 때

#### 실행 방법
```
/pm-execution:wwas SprintX "실시간 지출 추적기" --assumptions "모바일 우선, 예산 기능 필수"
```
제품명, 기능, 디자인 링크, 가정을 입력하면 WWA 형식의 백로그 아이템을 생성합니다.

#### 산출물 예시
```
Title: 실시간 지출 추적기 구현

Why: 사용자는 지출에 대한 즉각적 피드백이 필요하다. 이는 재정 인식
개선과 과소비 방지라는 목표에 직접 기여한다.

What: 사용자가 지출을 기록하면 실시간으로 업데이트되는 지출 추적기 추가.
설정된 예산 대비 현재 주간 지출을 표시. 디자인: [Figma 링크].
팀과의 대화를 위한 리마인더이며, 상세 스펙이 아님.

Acceptance Criteria:
- 지출 기록 후 2초 이내에 합계 업데이트
- 프로그레스 바로 예산 진행률 시각화
- 남은 예산 금액을 한눈에 확인 가능
- 여러 지출 카테고리를 올바르게 처리
```

#### 실전 팁
> 💡 WWA의 핵심은 "What"이 상세 스펙이 아니라 "대화를 위한 리마인더"라는 점입니다. 1~2문단 이내로 간결하게 유지하고, 상세 구현은 개발 과정에서 팀 대화로 결정합니다.

#### SprintX에서의 활용
AI 에이전트에게 작업을 지시할 때, "Why"(전략적 맥락)를 포함하면 에이전트가 더 나은 의사결정을 할 수 있습니다. 예를 들어 "Beachhead 세그먼트의 첫 경험을 개선하기 위해"라는 Why가 있으면, 에이전트가 관련 없는 엣지 케이스에 시간을 쓰지 않습니다.

#### 관련 스킬
- **이전**: create-prd (PRD를 WWA 아이템으로 분해)
- **다음**: test-scenarios (WWA에서 테스트 시나리오 파생)
- **대안**: user-stories (역할 중심), job-stories (상황 중심)

### 13. 테스트 시나리오 생성 (test-scenarios)
**난이도**: 중급
**예상 소요 시간**: 20~30분
**슬래시 명령어**: `/pm-execution:test-scenarios`

#### 이 스킬은 언제 쓰나요?
- 사용자 스토리의 인수 조건을 구체적인 QA 테스트 케이스로 변환할 때
- 기능 구현 전에 "어떻게 검증할 것인가"를 명확히 할 때
- 엣지 케이스와 오류 시나리오를 미리 식별할 때

#### 실행 방법
```
/pm-execution:test-scenarios SprintX "최근 본 항목 섹션" --user-story "온라인 쇼핑 사용자로서..."
```
사용자 스토리(제목 + 인수 조건)를 입력하면 종합적인 테스트 시나리오를 생성합니다.

#### 산출물 예시
```
Test Scenario: 제품 페이지에서 최근 본 제품 표시

Test Objective: "최근 본 항목" 섹션이 올바르게 표시되고
현재 제품이 제외되는지 검증

Starting Conditions:
- 사용자가 로그인했거나 브라우저 히스토리 활성화
- 현재 세션에서 2개 이상 제품 조회 완료

User Role: 온라인 쇼핑객

Test Steps:
1. 제품 페이지 이동 → 하단에 이전 조회 항목 표시
2. 페이지 하단 스크롤 → "최근 본 항목" 섹션 확인
3. 제품 썸네일 확인 → 이미지, 제목, 가격 표시
4. 현재 제품 확인 → 목록에 미포함
5. 카드 클릭 → 해당 제품 페이지로 이동

Expected Outcomes:
- 1개 이상 사전 조회 시에만 섹션 표시
- 4~8개 제품 카드 표시
- 각 카드에 "X분 전 조회" 타임스탬프
- 2초 이내 섹션 로딩
```

#### 실전 팁
> 💡 테스트 시나리오는 "행복한 경로(happy path)"뿐만 아니라 "불행한 경로(unhappy path)"도 포함해야 합니다. "아무 제품도 본 적 없는 사용자", "네트워크 끊김", "매우 많은 조회 기록" 같은 엣지 케이스를 반드시 고려하세요.

#### SprintX에서의 활용
user-stories에서 파생하여 QA 시나리오를 자동 생성합니다. AI 에이전트가 구현한 기능을 PO가 검증할 때, 이 테스트 시나리오를 체크리스트로 사용할 수 있습니다. SprintX에는 이미 326개 테스트가 있으므로, 새 기능의 테스트 시나리오도 기존 패턴에 맞춰 작성합니다.

#### 관련 스킬
- **이전**: user-stories, job-stories, wwas (스토리에서 테스트 시나리오 파생)
- **보완**: dummy-dataset (테스트에 필요한 데이터 생성)

### 14. 더미 데이터셋 생성 (dummy-dataset)
**난이도**: 초급
**예상 소요 시간**: 10~20분
**슬래시 명령어**: `/pm-execution:dummy-dataset`

#### 이 스킬은 언제 쓰나요?
- 프로토타이핑이나 개발 중 테스트 데이터가 필요할 때
- 데모용 현실적인 샘플 데이터를 만들어야 할 때
- 테스트 환경에 데이터를 채워야 할 때

#### 실행 방법
```
/pm-execution:dummy-dataset SprintX "고객 피드백" --rows 100 --format csv --columns "feedback_id, customer_name, email, rating, category, text"
```
데이터 유형, 행 수, 열 정의, 출력 형식, 비즈니스 규칙을 입력하면 즉시 사용 가능한 데이터를 생성합니다.

#### 산출물 예시
4가지 출력 형식 지원:
- **CSV**: 스프레드시트 가져오기용 플랫 테이블
- **JSON**: API와 NoSQL 데이터베이스용 중첩 구조
- **SQL**: 관계형 데이터베이스에 직접 실행 가능한 INSERT 문
- **Python 스크립트**: 대규모 또는 커스텀 데이터셋 생성용 실행 파일

비즈니스 규칙 적용 예시:
```
제약 조건:
- 평점 분포: 5점 40%, 4점 30%, 3점 20%, 1~2점 10%
- 버그 카테고리는 1~3점에만 연결
- 기능 요청은 3~5점에만 연결
- 이메일 도메인은 현실적으로 (gmail, yahoo 등)
```

#### 실전 팁
> 💡 더미 데이터를 만들 때 비즈니스 규칙을 반영하세요. "모든 평점이 균등 분포"인 데이터는 비현실적입니다. 실제 데이터의 편향(예: 대부분 4~5점)을 반영하면 더 유용한 테스트가 됩니다.

#### SprintX에서의 활용
프로토타이핑 시 테스트 데이터 생성에 활용합니다. 예를 들어 대시보드 개발 시 프로젝트, 스프린트, Goal, Task 데이터를 현실적으로 생성하여 UI가 다양한 상황에서 올바르게 작동하는지 검증할 수 있습니다.

#### 관련 스킬
- **보완**: test-scenarios (테스트 시나리오 실행에 필요한 데이터 생성)

### 15. 우선순위 프레임워크 가이드 (prioritization-frameworks)
**난이도**: 중급
**예상 소요 시간**: 30~45분
**슬래시 명령어**: `/pm-execution:prioritization-frameworks`

#### 이 스킬은 언제 쓰나요?
- 백로그 항목의 우선순위를 체계적으로 매기고 싶을 때
- RICE, ICE, Kano, MoSCoW 등 어떤 프레임워크를 써야 할지 모를 때
- 이해관계자에게 "왜 이것을 먼저 하는가"를 데이터 기반으로 설명해야 할 때

#### 실행 방법
```
/pm-execution:prioritization-frameworks SprintX 백로그 25개 이슈 우선순위 재배열
```
백로그 항목과 맥락을 입력하면, 적합한 프레임워크를 추천하고 각 항목의 점수를 계산합니다.

#### 산출물 예시
9개 프레임워크 중 상황에 맞는 것을 선택하여 적용합니다:

| 프레임워크 | 적합한 상황 | 핵심 인사이트 |
|-----------|-----------|-------------|
| Eisenhower Matrix | 개인 업무 관리 | 긴급 vs 중요 구분 |
| Impact vs Effort | 빠른 분류 | 간단한 2x2, 전략적 결정에는 부족 |
| Risk vs Reward | 불확실성 반영 | Impact vs Effort에 리스크 추가 |
| **Opportunity Score** | 고객 문제 우선순위 | 중요도 x (1 - 만족도). **권장** |
| Kano Model | 기대 수준 파악 | Must-be, Performance, Attractive 분류 |
| Weighted Decision Matrix | 다기준 의사결정 | 기준별 가중치 부여 |
| **ICE** | 아이디어/이니셔티브 | Impact x Confidence x Ease. **빠른 우선순위에 권장** |
| **RICE** | 대규모 아이디어 | (Reach x Impact x Confidence) / Effort |
| MoSCoW | 요구사항 분류 | Must/Should/Could/Won't |

핵심 원칙: **고객이 솔루션을 설계하게 하지 마세요. 기능이 아니라 문제(기회)에 우선순위를 매기세요.**

#### 실전 팁
> 💡 Opportunity Score(중요도 x (1 - 만족도))를 먼저 사용해서 고객 문제의 우선순위를 정한 후, ICE 또는 RICE로 구현 아이디어의 우선순위를 매기세요. "높은 중요도 + 낮은 만족도 = 가장 큰 기회"입니다.

#### SprintX에서의 활용
PO 가이드 **Phase 2 Step 2-1**에서 사용합니다. 25개 오픈 이슈(#161~#185)를 Phase 1의 전략 결과(NSM, Beachhead, Value Prop, 경쟁 분석, 가격 전략)에 맞춰 RICE/ICE로 재정렬합니다. 기존 P0~P3 라벨이 전략과 정렬되어 있지 않을 수 있으므로 전면 재평가합니다.

#### 관련 스킬
- **이전**: 전략 카테고리 스킬들 (전략이 우선순위의 기준)
- **다음**: brainstorm-okrs (정렬된 우선순위 기반 OKR 설정)
- **보완**: outcome-roadmap (우선순위에 따른 결과 기반 로드맵)
