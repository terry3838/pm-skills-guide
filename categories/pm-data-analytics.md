# 데이터 분석 (Data Analytics)

## 카테고리 소개

"직감으로 결정했는데 틀렸다"는 경험은 PM이라면 누구나 겪는다. 데이터 분석 카테고리는 직감을 증거로 바꾸는 도구 모음이다.

비개발자 PO가 데이터 분석을 어렵게 느끼는 이유는 주로 두 가지다. 첫째, SQL이나 Python 코드를 직접 작성해야 한다는 두려움. 둘째, 통계 용어(p-value, 신뢰구간 등)가 낯설다는 것. 이 카테고리의 스킬들은 그 두 장벽을 모두 AI가 처리하도록 설계되었다. PO는 "무엇을 알고 싶은지"를 말로 설명하면 된다.

세 스킬은 서로 보완 관계다. **SQL 쿼리 생성**은 원본 데이터를 꺼내오고, **코호트 분석**은 사용자 집단의 행동 패턴을 시간 축으로 추적하며, **A/B 테스트 분석**은 실험 결과를 통계적으로 검증해 "출시할지 말지"를 결정한다. 세 스킬을 함께 쓰면 "우리 온보딩의 어느 단계에서 이탈이 많은가 -> 이탈을 줄이는 변경을 실험 -> 실험이 유효한가"라는 데이터 기반 개선 루프를 완성할 수 있다.

## 이 카테고리의 스킬 한눈에 보기

| 스킬명 | 한 줄 설명 | 난이도 | 예상 소요 시간 |
|--------|-----------|--------|--------------|
| SQL 쿼리 생성 (sql-queries) | 말로 설명하면 SQL 쿼리로 변환. BigQuery, PostgreSQL, MySQL 등 지원 | 중급 | 10-15분 |
| 코호트 분석 (cohort-analysis) | 가입 시점별 사용자 집단의 리텐션과 기능 채택 패턴 분석 | 고급 | 30-45분 |
| A/B 테스트 분석 (ab-test-analysis) | 실험 결과의 통계적 유의성 검증과 출시/중단/연장 권고 | 고급 | 20-35분 |

## 추천 실행 순서

데이터 분석 스킬은 "데이터 수집 -> 패턴 발견 -> 실험 검증"의 순서로 실행한다.

![Diagram 1](../assets/diagrams/categories__pm-data-analytics__diagram_1.svg)

1. **SQL 쿼리 생성** (sql-queries) -- 분석에 필요한 데이터를 먼저 꺼낸다. Supabase(PostgreSQL)에서 사용자 행동 데이터를 추출하는 기초 단계다. 다른 두 스킬의 입력 데이터를 만드는 역할을 한다.
2. **코호트 분석** (cohort-analysis) -- SQL로 추출한 데이터를 시간 축으로 분석한다. 어떤 가입 cohort가 잘 남고 어떤 cohort가 이탈하는지 패턴을 찾는다.
3. **A/B 테스트 분석** (ab-test-analysis) -- 코호트 분석에서 발견한 문제를 개선하는 실험을 진행한 후, 결과가 통계적으로 유의미한지 판정한다.

> 사용자 데이터가 아직 없다면 1번만 실행해도 충분하다. 2번과 3번은 데이터가 쌓인 후에 의미가 있다.

## 관련 카테고리

- **Market Research**: 정성적 시장 조사를 정량적 데이터로 보완한다. `customer-journey-map`의 퍼널 단계를 `sql-queries`로 측정하고, `user-segmentation`의 세그먼트별 리텐션을 `cohort-analysis`로 추적한다.
- **Product Discovery**: 실험 설계는 `brainstorm-experiments`로, 결과 분석은 `ab-test-analysis`로 완성한다. 두 스킬은 함께 쓸 때 가장 효과적이다.
- **Execution**: `prioritization-frameworks`로 백로그를 정렬할 때 SQL 쿼리로 실제 사용 데이터를 근거로 제시할 수 있다.

**Step 3 — 산출물 검토**

생성된 쿼리를 Supabase Table Editor에서 실행하기 전에 다음을 확인한다:
- JOIN 방향이 맞는가? (LEFT JOIN vs INNER JOIN)
- WHERE 조건이 의도한 필터와 일치하는가?
- NULL 처리가 올바른가?
- 성능 주의사항(인덱스 권고)을 확인했는가?

#### 산출물 실제 예시
```sql
-- 지난 30일 가입 후 프로젝트 미생성 사용자 (집계)
SELECT
  COUNT(*) FILTER (WHERE p.id IS NULL) AS no_project_users,
  COUNT(*) AS total_new_users,
  ROUND(
    COUNT(*) FILTER (WHERE p.id IS NULL)::numeric / COUNT(*) * 100,
    1
  ) AS drop_off_rate_pct
FROM users u
LEFT JOIN (
  SELECT DISTINCT user_id FROM projects
) p ON p.user_id = u.id
WHERE u.created_at >= NOW() - INTERVAL '30 days'
  AND u.email NOT LIKE '%@example-company.com'
  AND u.email NOT LIKE '%@example.net'
  AND u.deleted_at IS NULL;

-- 결과 예시:
-- no_project_users | total_new_users | drop_off_rate_pct
-- 187              | 312             | 59.9

-- 사용자 목록 (이메일 타겟팅용)
SELECT
  u.id,
  u.email,
  u.created_at,
  MAX(ae.created_at) AS last_event_at
FROM users u
LEFT JOIN projects p ON p.user_id = u.id
LEFT JOIN analytics_events ae ON ae.user_id = u.id
WHERE u.created_at >= NOW() - INTERVAL '30 days'
  AND p.id IS NULL
  AND u.email NOT LIKE '%@example-company.com'
  AND u.deleted_at IS NULL
GROUP BY u.id, u.email, u.created_at
ORDER BY u.created_at DESC;

-- 성능 주의: users.created_at, projects.user_id에 인덱스 권장
-- CREATE INDEX IF NOT EXISTS idx_users_created_at ON users(created_at);
-- CREATE INDEX IF NOT EXISTS idx_projects_user_id ON projects(user_id);
```

### 2. 코호트 분석 (cohort-analysis)

> **난이도**: 고급 | **소요 시간**: 30-45분 | **명령어**: `/pm-data-analytics:cohort-analysis`

가입 시점별 사용자 집단의 리텐션과 기능 채택 패턴을 시간 축으로 추적해 "어느 cohort가 왜 남고 떠나는가"를 파악한다.

#### 실행 전 체크리스트
- [ ] 최소 3개 이상의 cohort 데이터가 있다 (각 cohort 최소 30명 이상 권장)
- [ ] 데이터가 CSV, Excel, 또는 JSON 형식으로 준비되어 있다
- [ ] 분석 목적(리텐션 개선? 기능 채택? Pro 전환?)이 명확하다
- [ ] cohort를 나누는 기준(가입 월? 온보딩 완료 여부?)을 결정했다

#### Step-by-Step 실행 가이드

**Step 1 — 명령어 실행**
```
/pm-data-analytics:cohort-analysis

파일: cohort_data.csv 첨부 (또는 아래 설명)
데이터 구조: cohort_month, user_id, weeks_active, feature_usage, subscription_tier
기간: 2026년 1월-3월 (3개 cohort)
분석 목표: 온보딩 개선(2월 적용) 전후 리텐션 변화 파악
```

**Step 3 — 산출물 검토**

코호트 분석 결과에서 다음을 확인한다:
- 리텐션 곡선이 예상 패턴(초기 급락 후 안정화)을 보이는가?
- cohort 간 유의미한 차이가 있는가? (단순 변동인지 추세인지)
- 이상 패턴(특정 주 리텐션 급증/급감)의 원인을 설명할 수 있는가?
- 후속 조치(인터뷰, A/B 테스트)를 위한 충분한 근거가 확보됐는가?

#### 산출물 실제 예시
```markdown
## 코호트 분석 결과: SprintX 1월-3월 2026

### 데이터 요약
- 분석 기간: 2026년 1월 1일 - 3월 31일
- 총 사용자: 247명 (1월 cohort: 82명, 2월: 91명, 3월: 74명)
- 데이터 품질: 이상값 3건 제거 후 244명 분석
- 리텐션 기준: 해당 주 최소 1개 Task 완료

### 주요 발견

**리텐션 곡선**
| Cohort | Week 1 | Week 2 | Week 4 | Week 8 |
|--------|--------|--------|--------|--------|
| 1월    | 68%    | 45%    | 31%    | 22%    |
| 2월    | 71%    | 51%    | 38%    | 27%    |
| 3월    | 75%    | 57%    | --     | --     |

**패턴 1: 온보딩 개선 효과 확인**
2월 cohort부터 Week 4 리텐션이 7%p 개선 (31% -> 38%).
성공 기준(36% 이상) 달성. 온보딩 개선이 효과를 낸 것으로 판단.

**패턴 2: Week 1-2 낙폭이 가장 큰 구간**
모든 cohort에서 Week 1-2 사이 낙폭이 평균 22%p.
초기 사용 경험 개선이 리텐션의 가장 큰 레버.

**이상 패턴: 1월 cohort Week 3 리텐션 소폭 반등**
1월 22일 Product Hunt 포스팅으로 재방문 사용자 유입.
트렌드 분석에서 제외하고 별도 주석 처리.

### 후속 조치 권고
1. (정량) Week 1-2 이탈 사용자 SQL 쿼리로 행동 패턴 추출
2. (정성) Week 2에 이탈한 1월 cohort 5명 대상 인터뷰 -- "무엇이 계속 쓰지 않게 만들었나"
3. (실험) 온보딩 체크리스트 추가를 A/B 테스트로 검증 (ab-test-analysis 스킬 사용)
4. (모니터링) 3월 cohort Week 4 리텐션을 4주 후 추적해 개선 지속 여부 확인
```

#### 다음 단계
- **A/B 테스트 분석**: 코호트에서 발견한 개선 가설을 실험으로 검증한다
- **SQL 쿼리 생성**: 이탈 사용자의 행동 패턴을 더 깊이 분석하는 쿼리를 만든다

### 3. A/B 테스트 분석 (ab-test-analysis)

> **난이도**: 고급 | **소요 시간**: 20-35분 | **명령어**: `/pm-data-analytics:ab-test-analysis`

실험 결과의 통계적 유의성을 검증하고 "출시/연장/중단" 중 하나를 명확한 근거와 함께 권고한다.

#### 실행 전 체크리스트
- [ ] 실험 가설이 명확하다 ("X를 바꾸면 Y가 Z만큼 개선된다")
- [ ] 1차 지표(Primary Metric)와 가드레일 지표가 정의되어 있다
- [ ] 실험 기간이 최소 1주일 이상이고 주간 사이클을 2회 포함한다
- [ ] 대조군/실험군 샘플 수와 전환 수가 준비되어 있다

#### Step-by-Step 실행 가이드

**Step 1 — 명령어 실행**
```
/pm-data-analytics:ab-test-analysis 온보딩 체크리스트 추가 실험

실험 내용: 온보딩 완료 후 체크리스트 위젯 추가 (변형 B)
가설: 체크리스트가 있으면 첫 Task 생성률이 높아진다
기간: 14일 (2026-02-20 ~ 2026-03-05)
트래픽 분할: 50/50
1차 지표: 7일 이내 첫 Task 생성률
가드레일 지표: 페이지 로드 시간, 전체 이탈률
결과: 대조군 1,240명 전환 312명 / 실험군 1,251명 전환 363명
```

**Step 2 — Claude의 질문과 답변 가이드**

---
**질문 1: [실험 설계 유효성]**
> Claude: "트래픽 분할이 50/50이고 샘플이 1,240 vs 1,251인데, 이 차이(0.9%)는 허용 범위 내인가요? 실험 시작 전에 샘플 크기 계산을 했나요?"

**✅ 좋은 답변 예시:**
```
실험 전 계산: 기대 효과 5%p, 기준 전환율 25%, 검정력 80%, 유의수준 5%로
계산했을 때 필요 샘플은 1,052명이었습니다.
실제 샘플(2,491명)이 충분합니다.
1,240 vs 1,251의 차이(0.9%)는 SRM(Sample Ratio Mismatch) 기준 5% 이내로 정상입니다.
```
**💡 팁:** SRM(샘플 비율 불일치)이 5% 초과하면 실험 자체가 오염된 것이다. 분할 로직에 버그가 있을 수 있다.

---
**질문 2: [가드레일 지표 설정 이유]**
> Claude: "페이지 로드 시간을 가드레일 지표로 선택한 이유가 있나요? 허용 가능한 성능 저하의 임계값은 어떻게 정했나요?"

**✅ 좋은 답변 예시:**
```
체크리스트 위젯이 추가 JavaScript 로딩을 요구하기 때문에 선택했습니다.
허용 임계값: 100ms 이내 증가. 그 이상이면 사용자 경험이 저하됩니다.
현재 기준 로드 시간이 1.2초이므로, 1.3초까지는 허용, 그 이상이면 최적화 필요.
```

---
**질문 3: [통계적 유의성 해석]**
> Claude: "p-value 0.003이 나왔는데, 이것이 정확히 무엇을 의미하는지, 비즈니스 맥락에서 어떻게 해석해야 하나요?"

**✅ 좋은 답변 예시:**
```
p-value 0.003: "이 결과가 우연일 확률이 0.3%"라는 의미입니다.
기준(p < 0.05)보다 훨씬 강한 유의성입니다.

비즈니스 해석: 체크리스트가 Task 생성률을 높인다는 증거가 매우 강합니다.
95% 신뢰구간으로 +5.4% ~ +24.8% 범위에서 효과가 있습니다.
최소 효과(+5.4%)도 비즈니스적으로 의미 있는 수준입니다.
```
**💡 팁:** p-value만 보지 말고 신뢰구간도 확인한다. 신뢰구간이 양수만 포함하면 방향이 확실하고, 0을 포함하면 불확실하다.

---
**질문 4: [가드레일 위반 처리]**
> Claude: "로드 시간이 통계적으로 유의미하게 100ms 증가했습니다. 이 경우 출시할지 말지 어떻게 결정해야 하나요?"

**✅ 좋은 답변 예시:**
```
출시 + 최적화 병행을 권고합니다.
근거:
1) 1차 지표(Task 생성률 +15.1%)의 비즈니스 가치가 100ms 저하보다 큼.
2) 100ms는 사용자가 인지하는 임계값(200ms) 미만.
3) 최적화로 해결 가능한 기술 이슈.

조건부 출시: 출시와 동시에 로드 시간 최적화 이슈를 백로그 최상단에 등록.
2주 내 100ms 개선 목표.
```

---
**질문 5: [장기 효과 추적]**
> Claude: "A/B 테스트가 14일 결과를 보여주는데, 장기 리텐션(30일, 60일)에도 효과가 유지되는지 어떻게 확인하나요?"

**✅ 좋은 답변 예시:**
```
전체 출시 후 코호트 분석으로 추적합니다.
- 체크리스트 배포 전(1월 cohort) vs 배포 후(3월 cohort)를 비교.
- 30일 리텐션과 Pro 전환율에 차이가 있는지 4주 후 cohort-analysis로 확인.
단기 A/B 테스트는 즉각적 효과를 보고, cohort 분석은 장기 효과를 검증합니다.
```
**💡 팁:** A/B 테스트는 단기 스냅샷이다. 장기 효과는 반드시 코호트 분석으로 별도 추적해야 한다.

---
**질문 6: [실험 종료 기준]**
> Claude: "만약 14일이 아니라 더 길게 실험을 해야 한다면 어떤 경우인가요? 언제 조기 종료해야 하나요?"

**✅ 좋은 답변 예시:**
```
연장 기준: 14일 후 p-value가 0.05 ~ 0.10 사이 (유의미하지만 약함). 1주 연장 후 재판정.
조기 종료 기준:
1) 가드레일 지표가 심각하게 악화 (이탈률 +10% 이상).
2) 1차 지표가 명확히 음수 방향 (조기 확인 후 중단 가능).
3) 기술적 버그로 실험 오염이 확인된 경우.
```

---

**Step 3 — 산출물 검토**

A/B 테스트 결과 문서에서 다음을 확인한다:
- SRM(샘플 비율 불일치)이 5% 이내인가?
- p-value 기준(0.05)을 충족하는가?
- 신뢰구간이 0을 포함하지 않는가?
- 가드레일 지표가 악화되지 않았는가?
- 권고(출시/연장/중단)에 명확한 근거가 있는가?

#### 산출물 실제 예시
```markdown
## A/B 테스트 결과: 온보딩 체크리스트 추가

**가설**: 체크리스트가 있으면 7일 이내 첫 Task 생성률이 높아진다
**기간**: 14일 | **샘플**: 대조군 1,240명 / 실험군 1,251명

### 테스트 설정 유효성 검증
- 샘플 크기: 충분 (기대 효과 5%p 기준 필요 샘플 1,052명, 실제 2,491명 ✅)
- 기간: 2주 = 주간 사이클 2회 포함. 적절함 ✅
- SRM (샘플 비율 불일치): 1,240 vs 1,251 (0.9% 차이). 정상 범위 ✅

### 통계 결과

| 지표 | 대조군 | 실험군 | 상승률 | p-value | 유의미? |
|------|--------|--------|--------|---------|---------|
| 첫 Task 생성률 (1차) | 25.2% | 29.0% | +15.1% | 0.003 | ✅ Yes |
| 전체 이탈률 (가드레일) | 41.3% | 40.8% | -1.2% | 0.61 | - 문제없음 |
| 로드 시간 (가드레일) | 1.2s | 1.3s | +8.3% | 0.02 | ⚠️ 모니터링 필요 |

95% 신뢰구간 (첫 Task 생성률): +5.4% ~ +24.8% (모두 양수 ✅)

### 권고: **출시 (Ship it) -- 조건부**

**근거**:
- 1차 지표(+15.1%)가 p=0.003으로 강한 유의미성 (기준: p<0.05) ✅
- 95% 신뢰구간 전체 양수 -- 효과 방향 확실 ✅
- 가드레일(이탈률) 악화 없음 ✅
- 로드 시간 100ms 증가는 사용자 인지 임계값(200ms) 미만, 최적화로 해결 가능 ⚠️

**다음 단계**:
1. 전체 트래픽에 체크리스트 배포
2. 로드 시간 최적화 이슈 백로그 최상단 등록 (2주 내 100ms 감소 목표)
3. 배포 후 4주 뒤 cohort-analysis로 30일 리텐션 효과 추적
```

#### 다음 단계
- **코호트 분석**: 출시 후 4주 뒤 장기 리텐션 효과를 cohort 단위로 추적한다
- **SQL 쿼리 생성**: 다음 실험을 위한 세그먼트별 데이터를 추출한다
