# 데이터 분석 (Data Analytics)

## 카테고리 소개

"직감으로 결정했는데 틀렸다"는 경험은 PM이라면 누구나 겪는다. 데이터 분석 카테고리는 직감을 증거로 바꾸는 도구 모음이다.

비개발자 PO가 데이터 분석을 어렵게 느끼는 이유는 주로 두 가지다. 첫째, SQL이나 Python 코드를 직접 작성해야 한다는 두려움. 둘째, 통계 용어(p-value, 신뢰구간 등)가 낯설다는 것. 이 카테고리의 스킬들은 그 두 장벽을 모두 AI가 처리하도록 설계되었다. PO는 "무엇을 알고 싶은지"를 말로 설명하면 된다.

세 스킬은 서로 보완 관계다. **SQL 쿼리 생성**은 원본 데이터를 꺼내오고, **코호트 분석**은 사용자 집단의 행동 패턴을 시간 축으로 추적하며, **A/B 테스트 분석**은 실험 결과를 통계적으로 검증해 "출시할지 말지"를 결정한다. 세 스킬을 함께 쓰면 "우리 온보딩의 어느 단계에서 이탈이 많은가 -> 이탈을 줄이는 변경을 실험 -> 실험이 유효한가"라는 데이터 기반 개선 루프를 완성할 수 있다.

## 이 카테고리의 스킬 한눈에 보기

| 스킬명 | 한 줄 설명 | 난이도 | 예상 소요 시간 |
|--------|-----------|--------|--------------|
| SQL 쿼리 생성 (sql-queries) | 말로 설명하면 SQL 쿼리로 변환. BigQuery, PostgreSQL, MySQL 등 지원 | 중급 | 10-15분 |
| 코호트 분석 (cohort-analysis) | 가입 시점별 사용자 집단의 리텐션과 기능 채택 패턴 분석 | 고급 | 30-45분 |
| A/B 테스트 분석 (ab-test-analysis) | 실험 결과의 통계적 유의성 검증과 출시/중단/연장 권고 | 고급 | 20-35분 |

## 추천 실행 순서

데이터 분석 스킬은 "데이터 수집 -> 패턴 발견 -> 실험 검증"의 순서로 실행한다.

1. **SQL 쿼리 생성** (sql-queries) -- 분석에 필요한 데이터를 먼저 꺼낸다. Supabase(PostgreSQL)에서 사용자 행동 데이터를 추출하는 기초 단계다. 다른 두 스킬의 입력 데이터를 만드는 역할을 한다.
2. **코호트 분석** (cohort-analysis) -- SQL로 추출한 데이터를 시간 축으로 분석한다. 어떤 가입 cohort가 잘 남고 어떤 cohort가 이탈하는지 패턴을 찾는다.
3. **A/B 테스트 분석** (ab-test-analysis) -- 코호트 분석에서 발견한 문제를 개선하는 실험을 진행한 후, 결과가 통계적으로 유의미한지 판정한다.

> 사용자 데이터가 아직 없다면 1번만 실행해도 충분하다. 2번과 3번은 데이터가 쌓인 후에 의미가 있다.

## 관련 카테고리

- **Market Research**: 정성적 시장 조사를 정량적 데이터로 보완한다. `customer-journey-map`의 퍼널 단계를 `sql-queries`로 측정하고, `user-segmentation`의 세그먼트별 리텐션을 `cohort-analysis`로 추적한다.
- **Product Discovery**: 실험 설계는 `brainstorm-experiments`로, 결과 분석은 `ab-test-analysis`로 완성한다. 두 스킬은 함께 쓸 때 가장 효과적이다.
- **Execution**: `prioritization-frameworks`로 백로그를 정렬할 때 SQL 쿼리로 실제 사용 데이터를 근거로 제시할 수 있다.

## 스킬 상세

### 1. SQL 쿼리 생성 (sql-queries)

**난이도**: 중급
**예상 소요 시간**: 10-15분
**슬래시 명령어**: `/pm-data-analytics:sql-queries`

#### 이 스킬은 언제 쓰나요?

- 시나리오 1: Supabase에서 "지난 30일간 가입했지만 첫 Task를 만들지 않은 사용자"를 뽑고 싶을 때
- 시나리오 2: 주간/월간 리포트를 위한 쿼리가 필요한데 SQL을 직접 작성하기 어려울 때
- 시나리오 3: 온보딩 퍼널의 각 단계별 전환율을 계산하고 싶을 때
- 시나리오 4: A/B 테스트나 코호트 분석을 위한 원본 데이터를 Supabase에서 추출할 때

#### 실행 방법

```
/pm-data-analytics:sql-queries

SprintX Supabase(PostgreSQL) 데이터베이스에서 다음 데이터를 추출해줘:
- 지난 30일간 가입했지만 첫 프로젝트를 만들지 않은 사용자 수와 비율
- 데이터베이스: users(id, email, created_at), projects(id, user_id, created_at)
- 방언: PostgreSQL
```

스키마 파일(`.sql` 또는 문서)이 있다면 함께 제공하면 더 정확한 쿼리가 생성된다.
SprintX는 `scripts/onboarding-funnel.sql`에 이미 8단계 퍼널 쿼리가 있으므로 이 파일을 참조로 제공하면 빠르게 시작할 수 있다.

#### 산출물 예시

```sql
-- 지난 30일 가입 후 프로젝트 미생성 사용자
SELECT
  COUNT(*) FILTER (WHERE p.id IS NULL) AS no_project_users,
  COUNT(*) AS total_new_users,
  ROUND(
    COUNT(*) FILTER (WHERE p.id IS NULL)::numeric / COUNT(*) * 100,
    1
  ) AS drop_off_rate_pct
FROM users u
LEFT JOIN projects p ON p.user_id = u.id
WHERE u.created_at >= NOW() - INTERVAL '30 days';

-- 설명: users 테이블과 projects 테이블을 LEFT JOIN하여
-- 프로젝트가 없는 사용자(p.id IS NULL)의 비율을 계산합니다.
-- Performance note: users.created_at, projects.user_id에 인덱스 권장
```

AI가 쿼리와 함께 아래를 제공한다:
- **쿼리 설명**: 각 JOIN, 조건, 집계가 무엇을 하는지 한국어로
- **성능 주의사항**: 대용량 테이블에서의 인덱스, 파티셔닝 권고
- **결과 검증 방법**: 쿼리가 맞는지 확인하는 방법

#### 실전 팁

> SQL을 전혀 모르더라도 괜찮다. "어떤 데이터를 알고 싶은지"를 구체적으로 말하는 것이 핵심이다. 막연하게 "사용자 분석 쿼리"보다 "지난 주에 가입해서 온보딩을 완료했지만 첫 Goal을 만들지 않은 사용자"처럼 조건을 명확히 할수록 더 정확한 쿼리가 나온다.

> Supabase의 Table Editor에서 직접 SQL을 실행할 수 있다. `scripts/onboarding-funnel.sql`처럼 자주 쓰는 쿼리는 파일로 저장해두면 재사용이 쉽다.

#### SprintX에서의 활용

SprintX는 이미 `scripts/onboarding-funnel.sql`에 8단계 온보딩 퍼널 쿼리를 보유하고 있다:
`landing_visited → login_started → onboarding_completed → dashboard_entered → project_created → first_sprint_created → first_goal_created → first_task_created`

이 퍼널의 각 단계별 전환율을 계산하는 쿼리를 SQL 쿼리 생성 스킬로 확장할 수 있다. Analytics 코드(`src/lib/analytics.ts`)의 13개 이벤트와 Supabase 스키마를 함께 제공하면 즉시 실전 쿼리를 만들 수 있다. PO 가이드 Phase 0.5에서 기존 자산 활용의 첫 단계로 실행하기 좋다.

#### 관련 스킬

- **보완**: `cohort-analysis` -- SQL로 추출한 데이터를 시간 축 분석에 입력한다
- **보완**: `ab-test-analysis` -- 실험 대조군/실험군 데이터를 SQL로 추출한 후 분석한다
- **연결**: `customer-journey-map` (Market Research) -- 여정 맵의 각 단계 전환율을 SQL로 측정한다

### 2. 코호트 분석 (cohort-analysis)

**난이도**: 고급
**예상 소요 시간**: 30-45분
**슬래시 명령어**: `/pm-data-analytics:cohort-analysis`

#### 이 스킬은 언제 쓰나요?

- 시나리오 1: "1월에 가입한 사용자와 2월에 가입한 사용자 중 누가 더 오래 남아 있나"를 비교하고 싶을 때
- 시나리오 2: 특정 기능 출시 이후 가입한 사용자들이 이전 사용자보다 리텐션이 좋아졌는지 확인하고 싶을 때
- 시나리오 3: 어떤 cohort가 Pro 플랜으로 전환하는 비율이 높은지 파악할 때
- 시나리오 4: "우리 제품의 리텐션이 개선되고 있는가, 악화되고 있는가"를 객관적으로 판단할 때

#### 실행 방법

```
/pm-data-analytics:cohort-analysis

파일: cohort_data.csv 첨부 (또는 아래 설명)
데이터 구조: cohort_month, user_id, weeks_active, feature_usage, subscription_tier
기간: 2026년 1월-3월 (3개 cohort)
분석 목표: 온보딩 개선(2월 적용) 전후 리텐션 변화 파악
```

CSV, Excel, JSON 형식의 데이터를 파일로 제공하거나 데이터 구조를 설명한다.
최소 3-4개 cohort가 있어야 패턴 파악이 의미 있다.

#### 산출물 예시

```
## 코호트 분석 결과: SprintX 1월-3월 2026

### 데이터 요약
- 분석 기간: 2026년 1월 1일 - 3월 31일
- 총 사용자: 247명 (1월 cohort: 82명, 2월: 91명, 3월: 74명)
- 데이터 품질: 이상값 3건 제거 후 244명 분석

### 주요 발견

**리텐션 곡선**:
| Cohort | Week 1 | Week 2 | Week 4 | Week 8 |
|--------|--------|--------|--------|--------|
| 1월    | 68%    | 45%    | 31%    | 22%    |
| 2월    | 71%    | 51%    | 38%    | 27%    |
| 3월    | 75%    | 57%    | --     | --     |

**패턴 1**: 2월 cohort부터 Week 4 리텐션이 7%p 개선.
온보딩 개선(2월 배포)이 효과를 낸 것으로 보임.

**패턴 2**: 모든 cohort에서 Week 1-2 사이 낙폭이 가장 큼 (평균 24%p).
초기 사용 경험 개선이 리텐션에 가장 큰 레버.

**이상 패턴**: 1월 cohort의 Week 3 리텐션이 Week 2보다 높음.
1월 말 새 기능 출시로 재방문한 사용자가 있는 것으로 추정.

### 후속 조치 권고
1. (정량) Week 1-2 이탈 사용자 대상 SQL 쿼리로 행동 패턴 분석
2. (정성) Week 2에 이탈한 1월 cohort 5명 대상 인터뷰 -- "무엇이 계속 쓰지 않게 만들었나"
3. (실험) 온보딩 체크리스트 개선을 A/B 테스트로 검증 제안
```

산출물에는 리텐션 히트맵, 코호트별 비교 차트, Python 재분석 스크립트(요청 시)가 포함된다.

#### 실전 팁

> 코호트를 나누는 기준을 명확히 정하는 것이 분석의 절반이다. 가입 월로 나눌 수도 있고, "온보딩 완료 여부", "첫 프로젝트 생성 여부", "Pro 전환 여부" 등 행동 기반으로도 나눌 수 있다. 무엇을 알고 싶은지에 따라 cohort 기준이 달라진다.

> 데이터가 적을 때(cohort당 50명 미만)는 숫자의 변동이 크다. 이 경우 방향성(개선/악화)만 참고하고, 정밀한 수치보다 "왜 그런지"를 인터뷰로 확인하는 것이 더 유용하다.

#### SprintX에서의 활용

PO 가이드 **Phase 3(격주 반복)** 에서 핵심적으로 사용한다. 격주 점검 루틴의 "실험 결과 리뷰"와 "OKR 체크인"에서, 코호트 분석으로 리텐션 추이를 추적한다.

구체적으로:
- 온보딩 개선(이슈 #161-#167 실험 적용) 전후 cohort를 비교해 "실험이 리텐션을 개선했나"를 측정한다
- Pro 전환 cohort와 Free 유지 cohort를 비교해 "어떤 행동이 유료 전환을 유도하는가"를 파악한다
- `scripts/onboarding-funnel.sql`로 추출한 데이터를 입력 파일로 활용한다

데이터는 Supabase에서 `sql-queries` 스킬로 추출한 후 CSV로 내보내 사용한다.

#### 관련 스킬

- **이전**: `sql-queries` -- 코호트 분석에 필요한 원본 데이터를 추출한다
- **다음**: `ab-test-analysis` -- 코호트에서 발견한 개선 가설을 실험으로 검증한다
- **연결**: `user-segmentation` (Market Research) -- 코호트 패턴과 세그먼트를 교차 분석한다
- **연결**: `sentiment-analysis` (Market Research) -- 리텐션이 낮은 cohort의 불만 요인을 정성적으로 파악한다

### 3. A/B 테스트 분석 (ab-test-analysis)

**난이도**: 고급
**예상 소요 시간**: 20-35분
**슬래시 명령어**: `/pm-data-analytics:ab-test-analysis`

#### 이 스킬은 언제 쓰나요?

- 시나리오 1: 온보딩 화면을 변경했는데 "이게 실제로 전환율을 높였는가"를 확인할 때
- 시나리오 2: 두 가지 랜딩 페이지 중 어떤 것이 더 많은 가입을 유도하는지 비교할 때
- 시나리오 3: 가격 변경 실험의 결과가 유의미한지, 아니면 그냥 우연의 일치인지 판단할 때
- 시나리오 4: 이슈 #161-#167 실험 결과를 보고 "출시할지, 더 실험할지, 중단할지"를 결정해야 할 때

#### 실행 방법

```
/pm-data-analytics:ab-test-analysis 온보딩 체크리스트 추가 실험

실험 내용: 온보딩 완료 후 체크리스트 위젯 추가 (변형 B)
가설: 체크리스트가 있으면 첫 Task 생성률이 높아진다
기간: 14일 (2026-02-20 ~ 2026-03-05)
트래픽 분할: 50/50
1차 지표: 7일 이내 첫 Task 생성률
가드레일 지표: 페이지 로드 시간, 전체 이탈률
결과: 대조군 1,240명 전환 312명 / 실험군 1,251명 전환 363명
```

결과 데이터(CSV 또는 숫자 직접 입력) 모두 가능하다.
데이터 파일을 첨부하면 Python으로 통계 계산을 자동으로 수행한다.

#### 산출물 예시

```
## A/B 테스트 결과: 온보딩 체크리스트 추가

**가설**: 체크리스트가 있으면 7일 이내 첫 Task 생성률이 높아진다
**기간**: 14일 | **샘플**: 대조군 1,240명 / 실험군 1,251명

### 테스트 설정 유효성 검증
- 샘플 크기: 충분 (기대 효과 크기 5%p 기준 필요 샘플 1,052명, 실제 2,491명)
- 기간: 2주 = 2회 이상의 주간 사이클 포함. 적절함.
- 샘플 비율 불일치(SRM): 1,240 vs 1,251 (0.9% 차이). 정상 범위.

### 통계 결과

| 지표 | 대조군 | 실험군 | 상승률 | p-value | 유의미? |
|------|--------|--------|--------|---------|---------|
| 첫 Task 생성률 (1차) | 25.2% | 29.0% | +15.1% | 0.003 | Yes |
| 전체 이탈률 (가드레일) | 41.3% | 40.8% | -1.2% | 0.61 | No |
| 로드 시간 (가드레일) | 1.2s | 1.3s | +8.3% | 0.02 | Yes* |

*로드 시간 100ms 증가는 통계적으로 유의미하나, 비즈니스적으로 허용 가능한 수준인지 검토 필요.

### 권고: **출시 (Ship it)**

**근거**:
- 1차 지표(+15.1%)가 p=0.003으로 강한 유의미성을 보임 (기준: p<0.05)
- 95% 신뢰구간: +5.4% ~ +24.8% (모두 양수)
- 가드레일(이탈률)은 악화 없음
- 로드 시간 100ms 증가는 모니터링 유지 권고

**다음 단계**:
1. 전체 트래픽에 체크리스트 배포
2. 로드 시간 최적화 이슈 백로그 등록
3. 배포 후 2주간 리텐션 지표 모니터링
```

#### 실전 팁

> A/B 테스트에서 가장 흔한 실수는 "결과가 좋아 보이면 바로 출시하는 것"이다. p-value를 확인하지 않으면 우연의 일치를 실제 효과로 착각할 수 있다. 반대로, p-value가 기준(0.05)을 아슬아슬하게 넘더라도 가드레일 지표가 악화되면 출시를 재고해야 한다. 이 스킬은 그 판단을 체계적으로 해준다.

> 테스트 기간이 너무 짧으면 신뢰할 수 없다. 최소 1-2주 이상, 주간 패턴을 2회 이상 포함해야 한다. 특히 사용자 수가 적을 때는 효과 크기를 현실적으로 설정해야 한다 (5%p 개선을 가정하면 수천 명이 필요하다).
