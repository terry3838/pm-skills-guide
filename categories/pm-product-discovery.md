# 제품 발견 (Product Discovery)

## 카테고리 소개

제품 발견은 "무엇을 만들어야 하는가"에 답하는 과정이다. 전략이 "어디로 갈 것인가"를 정한다면, 발견은 "어떤 길로 갈 것인가"를 찾는다. 코드를 짜기 전에, 혹은 이미 기능이 있는 제품에서도 반드시 거쳐야 할 질문들이 있다:

- **고객이 실제로 겪는 문제는 무엇인가?** (가정이 아니라 증거로)
- **우리가 만들려는 기능은 그 문제를 해결하는가?** (솔루션 편향 방지)
- **가장 위험한 가정부터 테스트하고 있는가?** (빠른 학습)
- **무엇을 만들지 않기로 결정했는가?** (초점 유지)

비개발자 PO에게 이 카테고리가 중요한 이유는 명확하다: **발견 없이 개발하면, AI 에이전트가 빠르게 만든 기능을 아무도 쓰지 않는 상황이 된다.** 속도가 빠를수록 잘못된 방향으로도 빠르게 달릴 수 있다.

이 카테고리의 13개 스킬은 Teresa Torres의 Continuous Discovery Habits, Alberto Savoia의 린 스타트업 실험, Dan Olsen의 Lean Product Playbook을 기반으로 설계되었다. 아이디어 발굴부터 사용자 인터뷰, 가정 검증, 기능 우선순위, 지표 설계까지 발견의 전 주기를 다룬다. 모두 Claude Code에서 슬래시 명령어로 실행할 수 있으며, 질문에 답하면 구조화된 발견 문서가 생성된다.

## 추천 실행 순서

제품 발견은 "문제에서 솔루션으로, 가정에서 증거로" 접근하는 것이 핵심이다. 아래 두 가지 흐름 중 상황에 맞게 선택한다.

### 흐름 A: 기존 제품에서 다음 기능을 결정할 때

![Diagram 1](../assets/diagrams/categories__pm-product-discovery__diagram_1.svg)

1. **인터뷰 스크립트** -- 고객에게 무엇을 물어볼지 먼저 준비한다
2. **인터뷰 요약** -- 인터뷰 결과를 구조화한다
3. **아이디어 브레인스토밍 (기존)** -- 인사이트를 기반으로 솔루션을 발산한다
4. **가정 식별 (기존)** -- 아이디어의 위험 요소를 드러낸다
5. **가정 우선순위화** -- 어떤 가정을 먼저 테스트할지 결정한다
6. **실험 설계 (기존)** -- 저비용으로 가정을 테스트하는 방법을 설계한다
7. **기능 우선순위화** 또는 **기능 요청 분석** -- 검증된 아이디어를 백로그로 정렬한다
8. **기회-솔루션 트리** -- 전체 발견 과정을 하나의 구조로 통합한다
9. **지표 대시보드 설계** -- "잘 되고 있는가"를 측정하는 체계를 만든다

### 흐름 B: 신규 제품 아이디어를 검증할 때

![Diagram 2](../assets/diagrams/categories__pm-product-discovery__diagram_2.svg)

> **팁**: 발견은 선형이 아니다. 실험이 실패하면 가정 식별로 돌아가고, 인터뷰에서 예상치 못한 인사이트가 나오면 아이디어 브레인스토밍을 다시 한다. Teresa Torres가 강조했듯, 발견은 한 번 하는 것이 아니라 매주 반복하는 습관이다.

## 관련 카테고리

| 카테고리 | 연결 이유 |
|----------|----------|
| **제품 전략 (Product Strategy)** | 전략 캔버스에서 도출된 "검증이 필요한 가정"이 발견의 출발점이 된다. `value-proposition`의 가정을 `brainstorm-experiments-existing`으로 테스트하고, `opportunity-solution-tree`로 전략과 발견을 연결한다 |
| **시장 조사 (Market Research)** | `customer-journey-map`은 기회-솔루션 트리의 기회 영역을 식별하는 데 필수다. `market-segments`로 발견 대상 고객을 구체화한다 |
| **실행 (Execution)** | 발견이 끝나면 `prioritization-frameworks`로 백로그를 정렬하고, `create-prd`로 개발 핸드오프한다. `outcome-roadmap`으로 발견 결과를 실행 계획으로 전환한다 |
| **Go-to-Market** | 인터뷰에서 얻은 고객 언어는 `gtm-strategy`의 메시지 작성에 직접 사용된다. Beachhead 세그먼트 발견은 `beachhead-segment` 스킬로 심화한다 |
| **마케팅 & 성장 (Marketing & Growth)** | `metrics-dashboard`에서 설계한 North Star Metric은 `north-star-metric` 스킬로 더 깊이 정의할 수 있다 |

## 스킬 상세

### 1. 아이디어 브레인스토밍 - 기존 제품 (brainstorm-ideas-existing)

> **난이도**: 초급 | **소요 시간**: 20-30분 | **명령어**: `/pm-product-discovery:brainstorm-ideas-existing`

PM·디자이너·엔지니어 3가지 시각으로 기존 제품의 기능 아이디어를 발굴하고 Top 5를 선정한다.

#### 실행 전 체크리스트
- [ ] 해결하려는 기회(고객 문제 또는 이탈 지점)가 명확히 정의되어 있다
- [ ] 목표 세그먼트(누구를 위한 아이디어인지)를 알고 있다
- [ ] 원하는 결과(이 아이디어들이 무엇을 달성해야 하는지)를 정했다
- [ ] 관련 인터뷰 결과나 데이터가 있으면 준비했다

#### Step-by-Step 실행 가이드

**Step 1 — 명령어 실행**
```
/pm-product-discovery:brainstorm-ideas-existing SprintX - AI 에이전트 기반 스프린트 관리 SaaS
```

**Step 2 — Claude의 질문과 답변 가이드**

---
**질문 1: 어떤 기회나 문제를 해결하려고 하나요?**
> Claude: "현재 제품에서 해결하려는 구체적인 기회 또는 문제를 설명해 주세요. 예: '사용자가 첫 Task 생성 후 7일 이내 이탈한다'"

**✅ 좋은 답변 예시:**
```
SprintX 사용자가 첫 Goal을 만든 뒤 7일 이내 이탈하는 문제.
온보딩 퍼널 SQL에서 first_goal_created 단계 이탈률이 62%로 가장 높다.
```
**💡 팁:** 데이터가 있으면 수치를 포함하면 아이디어의 방향이 더 구체적으로 나온다.

---
**질문 2: 이 기회에서 이익을 얻는 주요 고객 세그먼트는 누구인가요?**
> Claude: "아이디어의 대상이 되는 고객 세그먼트를 알려주세요."

**✅ 좋은 답변 예시:**
```
AI 코딩 에이전트(Claude Code, Cursor)를 주 3회 이상 사용하는 1인 개발자.
처음 SprintX를 접하고 첫 스프린트를 혼자 설정하는 단계에 있는 사용자.
```
**💡 팁:** 세그먼트가 좁을수록 아이디어가 더 실용적으로 나온다.

---
**질문 3: 이 기회에서 어떤 결과를 원하시나요?**
> Claude: "이 아이디어들이 달성해야 할 목표 결과는 무엇인가요? (예: 7일 유지율 10%p 개선)"

**✅ 좋은 답변 예시:**
```
7일 유지율을 현재 38%에서 50%로 개선.
사용자가 첫 스프린트를 완료하는 비율을 높이는 것이 핵심 지표.
```
**💡 팁:** 결과를 측정 가능한 지표로 표현하면 Top 5 선정 기준이 명확해진다.

---
**질문 4: 관련 연구 자료나 기회 트리가 있나요?**
> Claude: "인터뷰 결과, 데이터 분석, 고객 여정 맵 등 참고할 자료가 있으면 공유해 주세요."

**✅ 좋은 답변 예시:**
```
summarize-interview 산출물 첨부. 주요 인사이트: "어디서 시작해야 할지 모르겠다"는 피드백 3건.
고객 여정 맵의 'first_goal_created' 단계 끊김 지점 확인됨.
```
**💡 팁:** 없으면 "없음"이라고 답해도 된다. Claude가 가정으로 진행한다.

---
**질문 5: 현재 이미 시도해본 솔루션이 있나요?**
> Claude: "이미 구현했거나 실패한 솔루션이 있으면 알려주세요. 중복을 피할 수 있습니다."

**✅ 좋은 답변 예시:**
```
환영 이메일은 이미 있지만 클릭율 4%로 낮다.
튜토리얼 영상은 없음. 인앱 가이드도 없음.
```
**💡 팁:** 실패한 시도를 공유하면 같은 아이디어가 반복되지 않는다.

---

**Step 3 — 산출물 검토**

Claude가 PM 관점 5개, 디자이너 관점 5개, 엔지니어 관점 5개 아이디어를 생성한 뒤 Top 5를 선정한다. Top 5 표에서 "검증이 필요한 가정" 열을 확인하고 다음 스킬로 연결한다.

#### 산출물 실제 예시
```markdown
## 아이디어 브레인스토밍: SprintX 7일 이탈 개선

### Top 5 선정
| 순위 | 아이디어 | 선정 근거 | 검증이 필요한 가정 |
|------|---------|----------|-----------------|
| 1 | 온보딩 체크리스트 | 높은 임팩트, 낮은 개발 비용 | 사용자가 체크리스트를 따라간다 |
| 2 | 빈 상태(Empty State) 개선 | 즉시 구현 가능, 첫인상 개선 | 예시 데이터가 혼란을 주지 않는다 |
| 3 | AI 기반 첫 Sprint 자동 생성 | 차별화 높음 | AI 제안을 사용자가 신뢰한다 |
| 4 | 주간 진행률 이메일 다이제스트 | 재방문 유도 | 이메일을 실제로 읽는다 |
| 5 | 진행률 시각화 위젯 | 동기 부여 효과 | 숫자가 행동을 유발한다 |
```

#### 다음 단계
- **다음 스킬**: `identify-assumptions-existing` (Top 5의 가정을 4가지 리스크 영역에서 분석)

### 2. 아이디어 브레인스토밍 - 신규 제품 (brainstorm-ideas-new)

> **난이도**: 초급 | **소요 시간**: 20-30분 | **명령어**: `/pm-product-discovery:brainstorm-ideas-new`

새로운 제품의 핵심 기능 후보를 멀티 관점으로 발산하고 신규 제품 기준으로 Top 5를 선정한다.

#### 실행 전 체크리스트
- [ ] 제품 개념과 해결하려는 문제가 한 문장으로 정리되어 있다
- [ ] 타겟 고객 세그먼트를 정의했다
- [ ] 경쟁 제품이나 대체재를 파악하고 있다
- [ ] 비즈니스 모델(수익화 방향)을 어느 정도 생각해 두었다

#### Step-by-Step 실행 가이드

**Step 1 — 명령어 실행**
```
/pm-product-discovery:brainstorm-ideas-new SprintX - AI 에이전트 기반 스프린트 관리 SaaS
```

**Step 2 — Claude의 질문과 답변 가이드**

---
**질문 1: 제품 개념과 해결하려는 핵심 문제를 설명해 주세요.**
> Claude: "새로운 제품이 어떤 고객의 어떤 문제를 해결하나요?"

**✅ 좋은 답변 예시:**
```
AI 코딩 에이전트(Claude Code, Cursor)를 쓰는 개발자가 에이전트 작업 결과를 추적하고
리뷰하는 전용 프로젝트 관리 도구. 기존 Jira·Linear는 AI 에이전트 워크플로우를 지원하지 않음.
```
**💡 팁:** "누가, 어떤 상황에서, 무엇이 불편한가"를 포함하면 더 구체적인 아이디어가 나온다.

---
**질문 2: 타겟 고객 세그먼트는 누구인가요?**
> Claude: "초기 고객으로 집중할 세그먼트를 알려주세요."

**✅ 좋은 답변 예시:**
```
AI 코딩 에이전트를 주 3회 이상 사용하는 1인 개발자 또는 2-5인 소규모 팀.
주로 SaaS 사이드 프로젝트나 스타트업 초기 단계.
```
**💡 팁:** 세그먼트를 "모든 개발자"로 넓게 잡으면 아이디어가 분산된다. 좁게 시작하라.

---
**질문 3: 주요 경쟁자나 대체재는 무엇인가요?**
> Claude: "고객이 현재 이 문제를 어떻게 해결하고 있나요?"

**✅ 좋은 답변 예시:**
```
현재는 Notion 문서, GitHub Issues, 또는 스프레드시트로 수동 추적.
Linear·Jira는 AI 에이전트 이벤트 수신 기능이 없음.
```
**💡 팁:** 대체재를 알면 차별화 포인트가 있는 아이디어가 자동으로 부각된다.

---
**질문 4: MVP에 반드시 있어야 할 핵심 가치는 무엇인가요?**
> Claude: "이 제품의 존재 이유를 한 문장으로 표현한다면?"

**✅ 좋은 답변 예시:**
```
AI 에이전트가 무엇을 했는지 사람이 한눈에 파악하고 승인·반려할 수 있는 것.
추적과 통제가 핵심 가치.
```

**Step 3 — 산출물 검토**

4가지 리스크 영역(가치·사용성·실행가능성·실현가능성)별로 가정이 목록화되고 확신도(High/Medium/Low)와 테스트 방법이 제시된다. 확신도가 낮은 가정을 `prioritize-assumptions`으로 연결한다.

```markdown
### 실현 가능성 (Feasibility)
- Worker Daemon 없이 리뷰 플로우 독립 동작 가능 [확신도: 높음]

```

### 10. 기회-솔루션 트리 (opportunity-solution-tree)

> **난이도**: 고급 | **소요 시간**: 40-60분 | **명령어**: `/pm-product-discovery:opportunity-solution-tree`

목표 결과에서 출발해 기회·솔루션·실험을 계층 구조로 연결해 발견의 전체 지도를 만든다.

#### 실행 전 체크리스트
- [ ] 측정 가능한 목표 결과(OKR Key Result 또는 NSM)가 확정되어 있다
- [ ] 사용자 인터뷰 요약 또는 고객 여정 맵이 있다
- [ ] 기존에 식별한 기회(고객 문제)가 최소 2개 이상 있다
- [ ] 솔루션 아이디어를 하나로 좁히지 않고 여러 개 열어두고 있다

#### Step-by-Step 실행 가이드

**Step 1 — 명령어 실행**
```
/pm-product-discovery:opportunity-solution-tree SprintX - "7일 유지율 20% 달성"
```

**Step 2 — Claude의 질문과 답변 가이드**

---
**질문 1: 목표 결과 구체화**
> Claude: "목표 결과를 측정 가능한 형태로 표현해 주세요. 현재 수치도 알면 좋습니다."

**✅ 좋은 답변 예시:**
```
"7일 활성 사용자 중 Task를 3개 이상 완료한 비율 20%"
현재 수치는 측정 전이며 이번 달 기준선 측정 예정.
```
**💡 팁:** 현재 수치가 없으면 "측정 전"으로 입력해도 진행된다.

---
**질문 2: 기회 공간 입력**
> Claude: "인터뷰나 데이터에서 발견한 고객 문제(기회)를 고객의 언어로 말해주세요."

**✅ 좋은 답변 예시:**
```
"처음 접속했을 때 무엇을 해야 할지 모르겠다"
"AI가 무엇을 하고 있는지 알 수 없다"
"Task 완료 후 다음 단계가 자연스럽게 이어지지 않는다"
```
**💡 팁:** "기능이 없다"가 아니라 "나는 ...하기가 힘들다" 형식으로 작성하라.

---
**질문 3: 우선 탐색 기회**
> Claude: "이 기회들 중 가장 먼저 깊게 탐색하고 싶은 것은 무엇인가요?"

**✅ 좋은 답변 예시:**
```
기회 1("무엇을 해야 할지 모르겠다")과 기회 2("AI 실행 확인 불가")를
우선 탐색. 인터뷰 2건에서 모두 언급된 패턴이라 신뢰도가 높음.
```
**💡 팁:** 처음에는 1-2개만 깊게 파고, 나머지는 잠재 기회로 표시한다.

---
**질문 4: 솔루션 후보**
> Claude: "우선 기회에 대한 솔루션 아이디어를 3개 이상 말해주세요."

**✅ 좋은 답변 예시:**
```
기회 1 솔루션: A) 온보딩 체크리스트, B) 예시 프로젝트 자동 생성,
C) 인터랙티브 투어. 노력과 임팩트가 다르므로 비교 후 선택 예정.
```
**💡 팁:** 솔루션을 하나만 생각하면 "왜 이것인가"를 설명하기 어려워진다.

---
**질문 5: 실험 설계 범위**
> Claude: "각 솔루션에 대해 실험까지 설계할까요, 아니면 구조화만 할까요?"

**✅ 좋은 답변 예시:**
```
우선 기회 1의 솔루션 A에 대해서만 실험까지 설계.
나머지는 구조만 잡고 추후 brainstorm-experiments로 심화.
```
**💡 팁:** 처음 OST는 완벽하게 채우지 않아도 된다. 살아있는 문서로 관리한다.

---

**Step 3 — 산출물 검토**

목표 결과 → 기회 공간 → 솔루션 후보 → 실험 계층 구조가 연결되어 있는지, 각 솔루션에 가정과 성공 기준이 포함되어 있는지 확인하라.

#### 산출물 실제 예시
```markdown
## Opportunity Solution Tree: SprintX

### 목표 결과
"7일 활성 사용자 중 Task 3개 이상 완료 비율 20%"

### 기회 1: "처음 접속했을 때 무엇을 해야 할지 모르겠다"
솔루션 A: 온보딩 체크리스트 (낮은 노력, 중간 임팩트)
솔루션 B: 예시 프로젝트 자동 생성 (중간 노력, 높은 임팩트)
솔루션 C: 인터랙티브 투어 (높은 노력, 높은 임팩트)

  솔루션 A 실험:
  - 가정: 체크리스트를 보면 다음 단계를 따라간다
  - 방법: Feature stub으로 체크리스트 UI만 먼저 추가
  - 성공 기준: 7일 내 체크리스트 3단계 완료율 30% 이상
```

#### 다음 단계
- **다음 스킬**: `brainstorm-experiments-existing` (각 솔루션 실험 설계), `prioritize-features` (솔루션을 백로그로 정렬)

### 11. 인터뷰 스크립트 (interview-script)

> **난이도**: 초급 | **소요 시간**: 15-20분 | **명령어**: `/pm-product-discovery:interview-script`

JTBD 기반의 구조화된 고객 인터뷰 질문지와 노트 기록 템플릿을 생성한다.

- [ ] 인터뷰 대상 고객 유형이 정의되어 있다
- [ ] The Mom Test 원칙을 이해하고 있다 (과거 행동 질문, 피칭 금지)
- [ ] 인터뷰 일정과 채널(화상/대면/비동기)이 결정되어 있다

```
/pm-product-discovery:interview-script SprintX - AI 코딩 에이전트를 사용하는 개발자의 프로젝트 관리 방식과 불편함 탐색
```

**✅ 좋은 답변 예시:**
```
네, JTBD·현재 솔루션·핵심 고통·예상치 못한 발견·액션 아이템 항목.
인터뷰 직후 10분 안에 채울 수 있는 간결한 형식으로 요청.
```
**💡 팁:** 노트 템플릿은 summarize-interview 스킬의 입력 형식과 맞추면 연결이 쉬워진다.

---

**Step 3 — 산출물 검토**

여는 말·워밍업·핵심 탐색·마무리 구조가 있는지, 과거 행동 질문으로만 구성되어 있는지, 노트 기록 템플릿이 포함되어 있는지 확인하라.

#### 산출물 실제 예시
```markdown
## 인터뷰 스크립트: SprintX - AI 코딩 에이전트 사용자

### 여는 말
"오늘은 제 도구를 소개하는 것이 아니라 여러분이 일하는 방식을
이해하고 싶어서요. 맞는 답도 틀린 답도 없습니다."

### 핵심 질문 (5개)
1. "AI 코딩 에이전트를 마지막으로 쓴 게 언제인가요? 어떤 작업이었나요?"
2. "그 결과물을 어떻게 확인하셨나요? 어디에 기록하셨나요?"
3. "AI가 잘못된 코드를 만들었을 때 어떻게 하셨나요?"
4. "지금 프로젝트 진행 상황을 어떻게 관리하시나요?"
5. "그 부분에서 가장 힘든 게 뭔가요?"
```

#### 다음 단계
- **다음 스킬**: `summarize-interview` (인터뷰 완료 후 즉시 사용)

### 12. 인터뷰 요약 (summarize-interview)

> **난이도**: 중급 | **소요 시간**: 10-15분 | **명령어**: `/pm-product-discovery:summarize-interview`

인터뷰 녹취 전문을 JTBD·만족 신호·액션 아이템으로 구조화해 팀과 공유 가능한 1페이지 요약으로 변환한다.

#### 실행 전 체크리스트
- [ ] 인터뷰 녹취 전문 또는 상세 메모가 준비되어 있다
- [ ] 인터뷰 후 24시간 이내에 처리할 예정이다
- [ ] 참가자 이름/역할/배경 정보를 기록해 두었다
- [ ] 이전 인터뷰 요약이 있으면 패턴 비교를 위해 준비한다

#### Step-by-Step 실행 가이드

**Step 1 — 명령어 실행**
```
/pm-product-discovery:summarize-interview SprintX - AI 에이전트 기반 스프린트 관리 SaaS
[인터뷰 녹취 전문 또는 메모 붙여넣기]
```

**Step 2 — Claude의 질문과 답변 가이드**

---
**질문 1: 참가자 정보**
> Claude: "참가자의 역할, 경험, 현재 사용 중인 도구를 알려주세요."

**✅ 좋은 답변 예시:**
```
홍길동, 프리랜서 개발자 3년차, Claude Code 주 5회 사용.
Notion + GitHub Issues + 카카오톡 자기 자신에게 메모로 관리 중.
```
**💡 팁:** 배경이 명확할수록 인사이트의 대표성을 판단할 수 있다.

---
**질문 2: 인터뷰 목적**
> Claude: "이 인터뷰로 검증하려던 가정이나 주요 질문이 무엇이었나요?"

**✅ 좋은 답변 예시:**
```
"AI 결과 추적이 현재 불편하다"는 가정 검증.
지불 의향 신호가 나오면 별도로 표시해 달라고 요청.
```
**💡 팁:** 목적을 알면 Claude가 관련 내용에 가중치를 두고 요약한다.

---
**질문 3: 강조할 섹션**
> Claude: "요약에서 특히 강조했으면 하는 부분이 있나요?"

**✅ 좋은 답변 예시:**
```
핵심 인사이트에서 직접 인용문을 최대한 원문 그대로 보존해 달라.
액션 아이템은 날짜와 담당자까지 명시해 달라.
```
**💡 팁:** 고객의 실제 언어를 보존하면 나중에 랜딩 페이지 카피로 사용할 수 있다.

---
**질문 4: 패턴 비교**
> Claude: "이전 인터뷰 요약이 있으면 제공해 주세요. 패턴 비교를 포함할까요?"

**✅ 좋은 답변 예시:**
```
첫 번째 인터뷰라서 이전 데이터 없음.
3명 인터뷰 후 analyze-feature-requests로 통합 분석 예정.
```
**💡 팁:** 첫 인터뷰라도 요약을 표준 형식으로 저장하면 이후 비교가 쉬워진다.

---

**Step 3 — 산출물 검토**

현재 솔루션·좋은 점·문제점·핵심 인사이트(직접 인용 포함)·액션 아이템이 모두 포함되어 있는지, JTBD 형식으로 정리되어 있는지 확인하라.

#### 산출물 실제 예시
```markdown
## 인터뷰 요약: 홍길동 (프리랜서 개발자)

**참가자**: 홍길동, 3년차, Claude Code 주 5회 사용
**현재 솔루션**: Notion + GitHub Issues + 카카오톡 자기 메모

**현재 솔루션의 문제**:
- AI 만든 코드를 Notion에 수동 복사 (중요도: 높음, 만족도: 낮음)
- 어떤 AI 작업이 완료됐는지 추적 불가 (중요도: 높음, 만족도: 매우 낮음)

**핵심 인사이트**:
- "AI가 일을 다 했는데, 나는 내가 뭘 시켰는지 찾아야 해요"
- 지불 의향: "이게 자동으로 되면 월 10달러도 낼 것 같아요"

**액션 아이템**:
- [REDACTED_PHONE], PO: 5명 더 인터뷰해서 패턴 반복 여부 확인
```

#### 다음 단계
- **다음 스킬**: `analyze-feature-requests` (3건 이상 쌓이면 통합 분석), `opportunity-solution-tree` (기회 공간 업데이트)

### 13. 지표 대시보드 설계 (metrics-dashboard)

> **난이도**: 고급 | **소요 시간**: 40-60분 | **명령어**: `/pm-product-discovery:metrics-dashboard`

North Star Metric을 중심으로 Input Metrics·Health Metrics·Business Metrics 4계층 지표 체계와 대시보드 레이아웃을 설계한다.

#### 실행 전 체크리스트
- [ ] North Star Metric 또는 주요 비즈니스 목표가 확정되어 있다
- [ ] 분기 OKR 또는 목표 수치가 정해져 있다
- [ ] 현재 사용 중인 분석 도구(GA, Amplitude, Mixpanel 등)를 파악하고 있다
- [ ] 데이터 출처(DB 테이블명, 이벤트명)에 대한 기본 정보가 있다

#### Step-by-Step 실행 가이드

**Step 1 — 명령어 실행**
```
/pm-product-discovery:metrics-dashboard SprintX - North Star Metric: "주간 활성 사용자 중 Task를 완료한 비율", OKR: Q2 2026 첫 10명 유료 전환
```

**Step 2 — Claude의 질문과 답변 가이드**

---
**질문 1: NSM 정의 확인**
> Claude: "North Star Metric을 측정 가능한 공식으로 표현해 주세요. 현재 수치도 알려주세요."

**✅ 좋은 답변 예시:**
```
"가입일로부터 7일 내 Task 3개 이상 완료한 사용자 수 / 전체 신규 가입자 수"
현재: 측정 전. 이번 달 기준선 측정 예정. 목표: 20%.
```
**💡 팁:** 공식이 복잡하면 팀 공통 언어가 되지 않는다. 한 문장으로 설명 가능해야 한다.

---
**질문 2: Input Metrics 후보**
> Claude: "NSM을 움직이는 레버 지표 후보를 3-5개 말해주세요."

**✅ 좋은 답변 예시:**
```
온보딩 완료율, 첫 Goal 생성율, Task 생성→완료 전환율.
신규 가입자 수도 포함. 각각 NSM에 영향을 주는 퍼널 단계.
```
**💡 팁:** 각 Input Metric이 NSM과 어떻게 연결되는지 인과 관계를 설명하면 더 정확히 설계된다.

---
**질문 3: 데이터 출처**
> Claude: "각 지표를 어디서 가져올 수 있나요? DB 테이블이나 분석 이벤트가 있나요?"

**✅ 좋은 답변 예시:**
```
Supabase tasks 테이블 (status='done', created_at 기준).
analytics.ts의 13개 이벤트 중 onboarding_complete, first_goal_created.
```
**💡 팁:** 데이터 출처가 없으면 Claude가 "측정을 위해 추가해야 할 이벤트"도 함께 제안한다.

---
**질문 4: 알림 임계값**
> Claude: "어떤 수치 이하일 때 즉시 알림을 받고 싶으신가요?"

**✅ 좋은 답변 예시:**
```
NSM 10% 미만이면 즉시 알림. 온보딩 완료율 40% 미만도 포함.
에러율(API 5xx) 0.1% 초과 시 Slack 알림 설정 예정.
```
**💡 팁:** 알림 임계값이 있어야 대시보드가 매일 들여다보는 습관을 만든다.

---
**질문 5: 리뷰 주기**
> Claude: "지표를 매일/매주/매월 중 어떤 주기로 리뷰할 계획인가요?"

**✅ 좋은 답변 예시:**
```
매일: 에러율, 신규 가입자 (이상 감지용).
매주: Input Metrics, NSM 트렌드.
매월: Business Metrics (MRR, Churn), NSM 목표 대비 진행률.
```
**💡 팁:** 리뷰 주기에 따라 대시보드 레이아웃 우선순위가 달라진다.

---

**Step 3 — 산출물 검토**

4계층 지표 체계(NSM·Input·Health·Business)가 모두 정의되어 있는지, 각 지표에 데이터 출처·목표값·알림 임계값이 있는지, 대시보드 레이아웃과 리뷰 주기가 포함되어 있는지 확인하라.

#### 산출물 실제 예시
```markdown
## 제품 지표 대시보드 설계: SprintX

### North Star Metric
"7일 내 Task 3개 이상 완료한 신규 가입자 비율" — 목표: 20%
데이터 출처: Supabase tasks 테이블 / 알림: 10% 미만 시 즉시

### Input Metrics
| 지표 | 목표 | 알림 |
|------|------|------|
| 온보딩 완료율 | 60% | 40% 미만 |
| 첫 Goal 생성율 | 50% | 30% 미만 |
| Task 완료 전환율 | 40% | 20% 미만 |

### Health Metrics
- 에러율 (API 5xx): 0.1% 미만
- 페이지 로드 시간 (p95): 2초 미만

### Business Metrics
- MRR: $0 → $500 목표 (10명 × $5/월)
- Churn Rate: 월 5% 미만
```
