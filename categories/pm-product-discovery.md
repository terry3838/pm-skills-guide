# 제품 발견 (Product Discovery)

## 카테고리 소개

제품 발견은 "무엇을 만들어야 하는가"에 답하는 과정이다. 전략이 "어디로 갈 것인가"를 정한다면, 발견은 "어떤 길로 갈 것인가"를 찾는다. 코드를 짜기 전에, 혹은 이미 기능이 있는 제품에서도 반드시 거쳐야 할 질문들이 있다:

- **고객이 실제로 겪는 문제는 무엇인가?** (가정이 아니라 증거로)
- **우리가 만들려는 기능은 그 문제를 해결하는가?** (솔루션 편향 방지)
- **가장 위험한 가정부터 테스트하고 있는가?** (빠른 학습)
- **무엇을 만들지 않기로 결정했는가?** (초점 유지)

비개발자 PO에게 이 카테고리가 중요한 이유는 명확하다: **발견 없이 개발하면, AI 에이전트가 빠르게 만든 기능을 아무도 쓰지 않는 상황이 된다.** 속도가 빠를수록 잘못된 방향으로도 빠르게 달릴 수 있다.

이 카테고리의 13개 스킬은 Teresa Torres의 Continuous Discovery Habits, Alberto Savoia의 린 스타트업 실험, Dan Olsen의 Lean Product Playbook을 기반으로 설계되었다. 아이디어 발굴부터 사용자 인터뷰, 가정 검증, 기능 우선순위, 지표 설계까지 발견의 전 주기를 다룬다. 모두 Claude Code에서 슬래시 명령어로 실행할 수 있으며, 질문에 답하면 구조화된 발견 문서가 생성된다.

## 추천 실행 순서

제품 발견은 "문제에서 솔루션으로, 가정에서 증거로" 접근하는 것이 핵심이다. 아래 두 가지 흐름 중 상황에 맞게 선택한다.

### 흐름 A: 기존 제품에서 다음 기능을 결정할 때

```
인터뷰 스크립트 --> 사용자 인터뷰 진행 --> 인터뷰 요약
        |
        v
아이디어 브레인스토밍 (기존) --> 가정 식별 (기존) --> 가정 우선순위화
        |
        v
실험 설계 (기존) --> 실험 실행 --> 기능 우선순위화
        |
        v
기회-솔루션 트리 (전체 맥락 구조화) --> 지표 대시보드 설계
```

1. **인터뷰 스크립트** -- 고객에게 무엇을 물어볼지 먼저 준비한다
2. **인터뷰 요약** -- 인터뷰 결과를 구조화한다
3. **아이디어 브레인스토밍 (기존)** -- 인사이트를 기반으로 솔루션을 발산한다
4. **가정 식별 (기존)** -- 아이디어의 위험 요소를 드러낸다
5. **가정 우선순위화** -- 어떤 가정을 먼저 테스트할지 결정한다
6. **실험 설계 (기존)** -- 저비용으로 가정을 테스트하는 방법을 설계한다
7. **기능 우선순위화** 또는 **기능 요청 분석** -- 검증된 아이디어를 백로그로 정렬한다
8. **기회-솔루션 트리** -- 전체 발견 과정을 하나의 구조로 통합한다
9. **지표 대시보드 설계** -- "잘 되고 있는가"를 측정하는 체계를 만든다

### 흐름 B: 신규 제품 아이디어를 검증할 때

```
아이디어 브레인스토밍 (신규) --> 가정 식별 (신규) --> 가정 우선순위화
        |
        v
실험 설계 (신규 - XYZ 가설) --> 인터뷰 스크립트 --> 인터뷰 요약
        |
        v
기회-솔루션 트리 --> 지표 대시보드 설계
```

> **팁**: 발견은 선형이 아니다. 실험이 실패하면 가정 식별로 돌아가고, 인터뷰에서 예상치 못한 인사이트가 나오면 아이디어 브레인스토밍을 다시 한다. Teresa Torres가 강조했듯, 발견은 한 번 하는 것이 아니라 매주 반복하는 습관이다.

## 관련 카테고리

| 카테고리 | 연결 이유 |
|----------|----------|
| **제품 전략 (Product Strategy)** | 전략 캔버스에서 도출된 "검증이 필요한 가정"이 발견의 출발점이 된다. `value-proposition`의 가정을 `brainstorm-experiments-existing`으로 테스트하고, `opportunity-solution-tree`로 전략과 발견을 연결한다 |

| **실행 (Execution)** | 발견이 끝나면 `prioritization-frameworks`로 백로그를 정렬하고, `create-prd`로 개발 핸드오프한다. `outcome-roadmap`으로 발견 결과를 실행 계획으로 전환한다 |
| **Go-to-Market** | 인터뷰에서 얻은 고객 언어는 `gtm-strategy`의 메시지 작성에 직접 사용된다. Beachhead 세그먼트 발견은 `beachhead-segment` 스킬로 심화한다 |
| **마케팅 & 성장 (Marketing & Growth)** | `metrics-dashboard`에서 설계한 North Star Metric은 `north-star-metric` 스킬로 더 깊이 정의할 수 있다 |

## 스킬 상세

### 1. 아이디어 브레인스토밍 - 기존 제품 (brainstorm-ideas-existing)

**난이도**: 초급
**예상 소요 시간**: 20-30분
**슬래시 명령어**: `/pm-product-discovery:brainstorm-ideas-existing`

#### 이 스킬은 언제 쓰나요?

- **시나리오 1**: 다음 스프린트에 무엇을 만들지 막막할 때 -- "아이디어는 있는데 어떤 게 좋은지 모르겠다"
- **시나리오 2**: 특정 기회(예: "사용자 유지율이 낮다")를 발견했는데 어떤 솔루션이 있을지 탐색할 때
- **시나리오 3**: PM 혼자 생각하는 것이 아니라 디자이너·엔지니어 관점까지 포함한 다양한 아이디어를 원할 때

#### 실행 방법

```
/pm-product-discovery:brainstorm-ideas-existing [제품명 또는 기회 설명]
```

예시:
```
/pm-product-discovery:brainstorm-ideas-existing SprintX - 사용자가 첫 Task를 만든 후 7일 이내 이탈하는 문제
```

Claude가 아래 단계로 진행한다:
1. 기회와 목표 확인 (제품, 세그먼트, 원하는 결과)
2. PM 관점 5개, 디자이너 관점 5개, 엔지니어 관점 5개 아이디어 생성
3. 전략 정렬·잠재적 임팩트·실현 가능성·차별화를 기준으로 Top 5 선정

**입력으로 필요한 것**:
- 제품 설명과 현재 기회 또는 문제
- 목표 세그먼트 (누구를 위해 아이디어를 내는가)
- 원하는 결과 (이 아이디어들이 무엇을 달성해야 하는가)
- 관련 연구 파일이나 기회 트리가 있으면 같이 제공

#### 산출물 예시

```markdown
## 아이디어 브레인스토밍: SprintX 7일 이탈 개선

### PM 관점 아이디어 (5개)
1. 온보딩 체크리스트 -- 첫 Goal → Task → 완료까지 단계별 안내
2. 진행 상황 이메일 다이제스트 -- 주간 진행률 요약 자동 발송
...

### 디자이너 관점 아이디어 (5개)
1. 빈 상태(Empty State) 개선 -- 첫 접속 시 "시작하기" 예시 데이터 제공
...

### 엔지니어 관점 아이디어 (5개)
1. AI 기반 첫 Sprint 자동 생성 -- 프로젝트 설명을 입력하면 Sprint 구조를 자동으로 제안
...

### Top 5 선정

| 순위 | 아이디어 | 선정 근거 | 검증이 필요한 가정 |
|------|---------|----------|-----------------|
| 1 | 온보딩 체크리스트 | 높은 임팩트, 낮은 개발 비용, 이미 온보딩 퍼널 SQL 존재 | 사용자가 체크리스트를 따라간다 |
| 2 | 빈 상태 개선 | 즉시 구현 가능, 첫인상 개선 효과 | 예시 데이터가 혼란을 주지 않는다 |
...
```

#### 실전 팁

> Teresa Torres의 Product Trio 원칙: PM이 혼자 아이디어를 내는 것이 아니라 디자이너, 엔지니어와 함께 발산해야 한다. "최고의 아이디어는 종종 엔지니어에게서 나온다." 이 스킬은 그 3가지 관점을 Claude가 시뮬레이션해서 혼자서도 다양한 시각을 얻을 수 있게 한다.

> 기회-솔루션 트리(`opportunity-solution-tree`)와 함께 사용하면 더욱 강력하다. 아이디어를 내기 전에 먼저 "어떤 기회(고객 문제)를 해결하려는가"를 명확히 하면 아이디어의 질이 높아진다.

#### SprintX에서의 활용

PO 가이드의 **Phase 1 Step 1-6** 고객 여정 맵 완성 후, 끊김 지점이 식별된 구간에 대해 실행한다.

- **권장 입력**: 온보딩 퍼널 SQL(`scripts/onboarding-funnel.sql`)에서 이탈율이 높은 단계를 기회로 설정
- **산출물 활용**: Top 5 아이디어의 "검증이 필요한 가정"이 `brainstorm-experiments-existing` 스킬의 직접 입력이 됨
- **예시 기회**: "사용자가 `first_goal_created`에서 멈추는 이유가 뭔가" → 아이디어 발산 → 실험 설계

#### 관련 스킬

- **이전**: `customer-journey-map` (끊김 지점 식별), 사용자 인터뷰 결과 (`summarize-interview`)
- **다음**: `identify-assumptions-existing` (아이디어의 위험 가정 식별), `brainstorm-experiments-existing` (가정 테스트 설계)
- **보완**: `opportunity-solution-tree` (아이디어를 OST 구조로 통합), `prioritize-features` (아이디어를 백로그로 정렬)

### 2. 아이디어 브레인스토밍 - 신규 제품 (brainstorm-ideas-new)

**난이도**: 초급
**예상 소요 시간**: 20-30분
**슬래시 명령어**: `/pm-product-discovery:brainstorm-ideas-new`

#### 이 스킬은 언제 쓰나요?

- **시나리오 1**: 새로운 제품 아이디어가 있는데 어떤 핵심 기능부터 시작해야 할지 모를 때
- **시나리오 2**: MVP에 무엇을 넣고 무엇을 뺄지 기준이 필요할 때 -- "다 중요해 보여서 결정을 못 하겠다"
- **시나리오 3**: 스타트업 피칭 전에 기능 목록을 검증하고 싶을 때

#### 실행 방법

```
/pm-product-discovery:brainstorm-ideas-new [제품 개념 또는 설명]
```

예시:
```
/pm-product-discovery:brainstorm-ideas-new AI 코딩 에이전트의 작업 결과물을 추적하고 리뷰하는 프로젝트 관리 도구
```

신규 제품 발견이므로, 아이디어 선정 기준이 기존 제품과 다르다:
- **핵심 가치 전달 여부** -- 이것이 주요 문제를 해결하는가?
- **검증 속도** -- 빠르게 테스트할 수 있는가?
- **차별화 가능성** -- 경쟁자가 쉽게 복제할 수 없는가?

**입력으로 필요한 것**:
- 제품 개념과 목표 시장
- 타겟 고객 세그먼트
- 경쟁 환경 정보 (있으면)
- 시장 조사 파일 (있으면 같이 제공)

#### 산출물 예시

```markdown
## 아이디어 브레인스토밍: AI 에이전트 프로젝트 관리 도구 (신규)

### PM 관점 아이디어 (5개)
1. Goal-Task 계층 구조 -- AI 에이전트의 작업을 Goal 단위로 묶어 진행률 추적
2. 승인/반려 리뷰 플로우 -- AI가 만든 결과물을 사람이 검토하는 단계
...

### 디자이너 관점 아이디어 (5개)
1. 실행 로그 타임라인 -- AI가 무엇을 했는지 시간 순서로 시각화
...

### 엔지니어 관점 아이디어 (5개)
1. Webhook 기반 AI 이벤트 수신 -- 외부 AI 에이전트에서 실행 완료 이벤트를 받아 자동 기록
...

### Top 5 선정 (신규 제품 기준)

| 순위 | 아이디어 | 핵심 가치 전달 | 검증 속도 | 차별화 |
|------|---------|-------------|---------|-------|
| 1 | Goal-Task 계층 | 높음 | 빠름 | 중간 |
| 2 | 승인/반려 리뷰 | 높음 | 중간 | 높음 |
...
```

#### 실전 팁

> 신규 제품 발견은 "이 제품이 존재해야 하는가"를 묻는 것이다. 기능 아이디어를 발산하기 전에 먼저 "어떤 고객이 어떤 문제로 이 제품을 필요로 하는가"를 확인하자. 아이디어가 아무리 멋져도 고객 문제와 연결되지 않으면 의미가 없다.

> 이 스킬 직후에 `identify-assumptions-new`를 실행하면 각 아이디어의 위험 요소를 즉시 평가할 수 있다. 신규 제품에서는 8가지 리스크 카테고리(가치·사용성·실행 가능성·실현 가능성·윤리·GTM·전략·팀)를 모두 고려해야 한다.

#### SprintX에서의 활용

SprintX는 이미 기존 제품이므로 이 스킬보다 `brainstorm-ideas-existing`이 더 적합하다. 단, **완전히 새로운 기능 카테고리**(예: AI Provider Adapter, Mobile 앱)를 검토할 때는 신규 제품과 같은 사고방식으로 접근하는 것이 유용하다. `lean-canvas` 스킬과 함께 사용하면 새 기능의 비즈니스 케이스를 빠르게 만들 수 있다.

#### 관련 스킬

- **이전**: `product-strategy` (전략 방향 설정 후), `startup-canvas` 또는 `lean-canvas` (비즈니스 케이스와 연계)
- **다음**: `identify-assumptions-new` (신규 제품의 8가지 리스크 분석), `brainstorm-experiments-new` (XYZ 가설로 개념 검증)
- **보완**: `opportunity-solution-tree` (아이디어를 OST로 구조화), `prioritize-features` (Top 5 아이디어 백로그화)

### 3. 실험 설계 - 기존 제품 (brainstorm-experiments-existing)

**난이도**: 중급
**예상 소요 시간**: 25-35분
**슬래시 명령어**: `/pm-product-discovery:brainstorm-experiments-existing`

#### 이 스킬은 언제 쓰나요?

- **시나리오 1**: 기능을 개발하기 전에 "이게 정말 필요한가"를 저비용으로 테스트하고 싶을 때
- **시나리오 2**: 가정 목록이 있는데 어떤 방법으로 테스트할지 모를 때 -- "A/B 테스트? 사용자 인터뷰? 뭐가 맞나?"
- **시나리오 3**: 개발 비용을 줄이고 싶을 때 -- "만들기 전에 먼저 원하는지 확인하자"

#### 실행 방법

```
/pm-product-discovery:brainstorm-experiments-existing [제품명과 검증하려는 기능/가정]
```

예시:
```
/pm-product-discovery:brainstorm-experiments-existing SprintX - "사용자가 AI 실행 결과를 검토하는 리뷰 플로우를 실제로 사용할 것이다"라는 가정 검증
```

실험 방법 후보로는 다음과 같은 것을 제안한다:
- **First-click 테스트 또는 프로토타입** -- 사용자가 UI를 직관적으로 이해하는지 확인
- **Feature stub (가짜 문 테스트)** -- 기능을 구현하지 않고 버튼만 만들어 클릭율 측정
- **Technical spike** -- 기술적 실현 가능성 검증
- **A/B 테스트** -- 프로덕션에서 두 버전 비교 (위험 완화 방법 포함)
- **Wizard of Oz** -- 자동화 없이 수동으로 서비스를 제공해 수요 확인
- **설문 기반 검증** -- 의견이 아닌 행동 기반으로 설계

**입력으로 필요한 것**:
- 기능 아이디어 설명
- 검증하려는 구체적 가정 목록 (있으면 `identify-assumptions-existing` 산출물 사용)
- PRD, 디자인 파일 등 관련 자료

#### 산출물 예시

```markdown
## 실험 설계: SprintX 리뷰 플로우

| 가정 | 실험 방법 | 측정 지표 | 성공 기준 |
|------|---------|---------|---------|
| 사용자가 리뷰 플로우를 사용한다 | Feature stub: "AI 결과 검토" 버튼을 먼저 추가, 실제 기능 없이 클릭율 측정 | 버튼 클릭율 (GA 이벤트) | 7일 내 활성 사용자 중 20% 이상 클릭 |
| 승인/반려 플로우가 직관적이다 | First-click 테스트: Figma 프로토타입으로 5명 테스트 | 첫 클릭 정확도 | 80% 이상이 첫 클릭에서 올바른 액션 수행 |
| 리뷰 없이 사용하는 사람도 있다 | A/B 테스트: 리뷰 강제 vs 선택적 플로우 | 7일 유지율 | 강제 리뷰 그룹의 유지율이 5%p 이상 낮지 않을 것 |
```

#### 실전 팁

> 핵심 원칙: "의견이 아니라 행동을 측정하라." "이 기능을 쓸 것 같아요?"라는 질문은 무의미하다. 실제로 버튼을 클릭했는가, 실제로 구매했는가만이 신뢰할 수 있는 데이터다.

> 실험 전에 반드시 성공 기준(Success Threshold)을 먼저 정하라. "실험을 해보고 결과를 보겠다"는 사후 합리화의 함정이다. 30% 클릭율이 나왔을 때 성공인지 실패인지를 미리 결정해야 객관적으로 판단할 수 있다.

#### SprintX에서의 활용

PO 가이드 **Phase 1 Step 1-6**에서 고객 여정 맵 직후에 실행하는 핵심 스킬이다. 기존 이슈 #161-#167(EXP 실험 시리즈)과 직접 연결된다.

- **#161 Fake Door Test**: `feature-stub` 실험 방법 적용 -- 아직 없는 기능에 버튼만 추가해 수요 측정
- **#163 Wizard of Oz**: 수동으로 AI 실행을 시뮬레이션해 사용자 반응 확인 (Worker daemon 개발 전)
- **#165 가격 민감도**: A/B 테스트 또는 Van Westendorp 설문 설계
- 온보딩 퍼널 SQL의 각 단계별 이탈에 대한 실험을 설계하면 #166 실험과 연동됨

#### 관련 스킬

- **이전**: `identify-assumptions-existing` (가정 목록), `prioritize-assumptions` (어떤 가정을 먼저 테스트할지)
- **다음**: 실험 실행 후 → `summarize-interview` (인터뷰 데이터 처리), `prioritize-features` (검증 결과 기반 백로그 정렬)
- **보완**: `opportunity-solution-tree` (실험을 OST의 4번째 레벨로 통합), `customer-journey-map` (어떤 단계에서 실험할지 결정)

### 4. 실험 설계 - 신규 제품 (brainstorm-experiments-new)

**난이도**: 중급
**예상 소요 시간**: 20-30분
**슬래시 명령어**: `/pm-product-discovery:brainstorm-experiments-new`

#### 이 스킬은 언제 쓰나요?

- **시나리오 1**: 새로운 제품 아이디어의 시장 수요를 코드 한 줄 짜기 전에 확인하고 싶을 때
- **시나리오 2**: "이 아이디어가 괜찮을 것 같다"는 직감을 데이터로 바꾸고 싶을 때
- **시나리오 3**: 제한된 자원으로 여러 아이디어 중 가장 가능성 있는 것을 빠르게 선별할 때

#### 실행 방법

```
/pm-product-discovery:brainstorm-experiments-new [제품 개념]
```

예시:
```
/pm-product-discovery:brainstorm-experiments-new AI 코딩 에이전트 전용 태스크 관리 SaaS
```

Claude가 아래 단계로 진행한다:
1. **XYZ 가설 작성**: "Y 중 최소 X%가 Z를 할 것이다" 형식 (예: "AI 에이전트를 쓰는 개발자 중 15%가 30일 무료 체험을 신청할 것이다")
2. **Pretotype 실험 2-3개 제안**: 랜딩 페이지, 설명 영상, 이메일 캠페인, 사전 주문, 수동 MVP 등
3. **각 실험의 가설·방법·측정 지표·성공 기준 명시**

**핵심 원칙** (Alberto Savoia, *The Right It*):
- **Skin-in-the-Game**: 관심이 아니라 지불 의향을 테스트하라. 실제 시간, 돈, 평판을 거는 행동만이 신뢰할 수 있는 신호다
- **YODA (Your Own Data)**: 남의 데이터(시장 보고서, 유사 사례)가 아니라 직접 수집한 데이터를 사용하라

**입력으로 필요한 것**:
- 제품 개념 설명
- 타겟 시장 정의
- 시장 조사 파일 또는 랜딩 페이지 초안 (있으면)

#### 산출물 예시

```markdown
## 린 스타트업 실험 설계: AI 에이전트 태스크 관리 SaaS

### XYZ 가설
"AI 코딩 에이전트(Claude Code, Cursor)를 주 3회 이상 사용하는 개발자 중
**최소 10%**가 SprintX 30일 무료 체험을 신청하고
**최소 3%**가 Pro $5/월로 전환할 것이다."

### 실험 1: 랜딩 페이지 + 대기자 명단
- **방법**: 제품 없이 가치 제안만 있는 랜딩 페이지, 이메일 등록 버튼
- **측정 지표**: 방문자 대비 등록율 (목표: 5% 이상)
- **실행 방법**: Hacker News, Reddit r/programming에 포스팅, Twitter/X 광고 $100

### 실험 2: 수동 MVP (Wizard of Oz)
- **방법**: 자동화 없이 스프레드시트+슬랙으로 5명에게 수동으로 서비스 제공
- **측정 지표**: 3주 후 계속 사용 의향 여부, 지불 의향 금액
- **Skin-in-the-Game**: "다음 달부터 $5/월을 내겠다"는 확인을 받아야 성공

### 결론: 어떤 신호가 나오면 계속 개발하는가
- 등록율 5% 이상 + 수동 MVP 고객 중 3명 이상 지불 의향 표명
```

#### 실전 팁

> XYZ 가설을 작성할 때 숫자가 막막하다면 역산으로 시작하라. "첫 달에 유료 고객 10명이 필요하다 → 전환율 5%라면 200명의 체험 사용자 → 랜딩 페이지 방문자 2000명이 필요하다" 이렇게 역산하면 현실적인 X% 값이 나온다.

> "이 실험 결과가 어떻게 나와도 계속 개발할 것 같다"는 느낌이 든다면, 그 실험은 제대로 된 실험이 아니다. 좋은 실험은 "이 결과가 나오면 방향을 바꾸겠다"는 조건을 미리 설정해 두는 것이다.

#### SprintX에서의 활용

SprintX는 이미 기술 MVP가 있으므로 신규 제품 실험보다는 기존 제품 실험(`brainstorm-experiments-existing`)이 더 적합하다. 단, **완전히 새로운 수익화 아이디어**(예: AI Provider 마켓플레이스, 에이전트 템플릿 판매)를 검토할 때 XYZ 가설 방식으로 수요를 먼저 확인하는 데 활용할 수 있다. 기존 Waitlist API와 연동하면 즉시 Skin-in-the-Game 테스트가 가능하다.

#### 관련 스킬

- **이전**: `brainstorm-ideas-new` (아이디어 발산 후 검증 설계), `identify-assumptions-new` (위험 가정 식별 후)
- **다음**: `summarize-interview` (수동 MVP 고객 인터뷰 정리), `prioritize-features` (검증 결과 기반 기능 선정)
- **보완**: `lean-canvas` (실험 결과를 비즈니스 모델로 통합), `startup-canvas` (신규 제품의 전략과 연계)

### 5. 가정 식별 - 기존 제품 (identify-assumptions-existing)

**난이도**: 초급
**예상 소요 시간**: 20-25분
**슬래시 명령어**: `/pm-product-discovery:identify-assumptions-existing`

#### 이 스킬은 언제 쓰나요?

- **시나리오 1**: 기능 아이디어가 있는데 "이게 실패할 수 있는 이유가 뭘까"를 체계적으로 따져보고 싶을 때
- **시나리오 2**: PRD를 작성하기 전에 미리 위험 요소를 드러내고 싶을 때
- **시나리오 3**: 개발을 시작했는데 뭔가 걱정되는 부분이 있을 때 -- "내가 놓치고 있는 게 있을까?"

#### 실행 방법

```
/pm-product-discovery:identify-assumptions-existing [제품명 + 기능 아이디어]
```

예시:
```
/pm-product-discovery:identify-assumptions-existing SprintX - "AI 실행 결과 승인/반려 플로우" 기능
```

Claude가 3가지 관점(PM·디자이너·엔지니어)에서 실패 가능성을 분석하고, 4가지 리스크 영역에서 가정을 도출한다:

- **가치 (Value)**: 고객에게 실제 가치를 주는가? 진짜 문제를 해결하는가?
- **사용성 (Usability)**: 사용자가 직관적으로 쓸 수 있는가? 학습 곡선이 적절한가?
- **실행 가능성 (Viability)**: 마케팅, 영업, 재무, 법무가 지원할 수 있는가?
- **실현 가능성 (Feasibility)**: 현재 기술로 만들 수 있는가? 통합 리스크는?

**입력으로 필요한 것**:
- 제품 설명과 기능 아이디어
- 관련 디자인·PRD 파일 (있으면)

#### 산출물 예시

```markdown
## 위험 가정 분석: SprintX 승인/반려 플로우

### 가치 (Value) 리스크
- **가정**: 개발자들이 AI 결과물을 검토하는 데 시간을 쓸 의향이 있다
  - 문제 가능성: AI 신뢰도가 높아질수록 검토 자체가 불필요해질 수 있음
  - 확신도: 중간
  - 테스트 방법: 인터뷰에서 "지금 AI 결과물을 어떻게 검토하나요?"를 직접 관찰

### 사용성 (Usability) 리스크
- **가정**: "승인"과 "반려"의 구분이 사용자에게 직관적이다
  - 문제 가능성: "반려 후 어떻게 되는가"를 이해 못 할 수 있음
  - 확신도: 낮음
  - 테스트 방법: 프로토타입 First-click 테스트 (5명)

### 실행 가능성 (Viability) 리스크
- **가정**: Free 플랜 사용자도 리뷰 플로우를 사용한다
  - 문제 가능성: AI 실행이 Pro 기능이라면 Free 사용자는 리뷰할 것이 없음
  - 확신도: 높음
  - 테스트 방법: 가격 전략 재검토 필요 (pricing-strategy 스킬)

### 실현 가능성 (Feasibility) 리스크
- **가정**: Worker daemon 없이도 리뷰 플로우는 독립적으로 동작한다
  - 확신도: 높음 (기술 검토 완료)
  - 테스트 방법: N/A (확신도가 높아 별도 실험 불필요)
```

#### 실전 팁

> 이 스킬은 "악마의 대변인" 역할이다. 기능을 죽이는 것이 목적이 아니라, 강하게 만드는 것이 목적이다. 발견된 가정은 위험 요소를 제거하는 실험 설계로 이어진다.

> 확신도(High/Medium/Low)를 정할 때 실용적 기준을 쓰자: 데이터나 증거가 있으면 High, 비슷한 사례가 있으면 Medium, 순수 가정이면 Low. Low인 가정일수록 먼저 테스트해야 한다.

#### SprintX에서의 활용

이슈 백로그(#168-#185)의 각 기능에 대해 개발 시작 전에 실행하면, 가장 위험한 가정을 드러낼 수 있다. 특히:

- **#168 Worker Daemon**: "개발자들이 AI 자동 실행을 원한다"는 가정이 검증되었는가?
- **#184 Dark Mode**: "Dark Mode가 Beachhead 세그먼트에게 중요한 가치인가?" vs 단순 선호 기능인가?
- Phase 0에서 분류한 Freeze 이슈들을 해제하기 전에 이 스킬로 가정을 먼저 식별한다

#### 관련 스킬

- **이전**: `brainstorm-ideas-existing` (아이디어 발산 후), `brainstorm-ideas-new` (신규 아이디어 평가)
- **다음**: `prioritize-assumptions` (어떤 가정을 먼저 테스트할지), `brainstorm-experiments-existing` (실험 설계)
- **보완**: `swot-analysis` (위험 요소를 내/외부 요인으로 분류), `identify-assumptions-new` (신규 제품이라면 이것 사용)

### 6. 가정 식별 - 신규 제품 (identify-assumptions-new)

**난이도**: 초급
**예상 소요 시간**: 25-35분
**슬래시 명령어**: `/pm-product-discovery:identify-assumptions-new`

#### 이 스킬은 언제 쓰나요?

- **시나리오 1**: 새로운 제품 아이디어가 있는데 "어디서 실패할 수 있는가"를 전방위적으로 파악하고 싶을 때
- **시나리오 2**: 기존 제품의 리스크 분석(4가지)이 부족하고, GTM·전략·팀 리스크까지 포함하고 싶을 때
- **시나리오 3**: 투자자나 이해관계자에게 "우리가 위험을 알고 있다"는 것을 보여주고 싶을 때

#### 실행 방법

```
/pm-product-discovery:identify-assumptions-new [제품 개념 또는 설명]
```

예시:
```
/pm-product-discovery:identify-assumptions-new AI 코딩 에이전트 전용 프로젝트 관리 SaaS - 개발자 1인 또는 소규모 팀 타겟
```

기존 제품 스킬이 4가지 리스크를 분석하는 반면, 신규 제품 스킬은 **8가지 리스크 카테고리**를 다룬다:

1. **가치 (Value)**: 고객에게 가치를 주는가?
2. **사용성 (Usability)**: 사용자가 쓸 수 있는가?
3. **실행 가능성 (Viability)**: 수익화·법률·확장이 가능한가?
4. **실현 가능성 (Feasibility)**: 기술적으로 만들 수 있는가?
5. **윤리 (Ethics)**: 만들어도 되는가? 사용자에게 해가 되지 않는가?
6. **GTM (Go-to-Market)**: 마케팅할 수 있는가? 채널이 있는가?
7. **전략 & 목표 (Strategy & Objectives)**: 경쟁자가 복제할 수 있는가? 거시 환경 변화는?
8. **팀 (Team)**: 우리 팀이 이것을 실행할 수 있는가?

**입력으로 필요한 것**:
- 제품 또는 스타트업 아이디어 설명
- 타겟 시장과 고객 세그먼트
- 사업 계획이나 연구 파일 (있으면)

#### 산출물 예시

```markdown
## 신규 제품 위험 가정 분석 (8가지 카테고리)

### 1. 가치 (Value)
- 개발자들이 AI 에이전트 결과물 추적에 충분한 고통을 느끼고 있다 [확신도: 낮음]
- Goal-Task 계층 구조가 현재 수동 방식보다 실질적으로 빠르다 [확신도: 중간]

### 5. 윤리 (Ethics)
- AI 에이전트의 코드 실행 로그를 저장하는 것이 사용자 동의 하에 이루어진다 [확신도: 높음]

### 6. GTM
- "AI 코딩 에이전트 사용자"라는 세그먼트에 도달하는 채널이 존재한다 [확신도: 낮음]
- Hacker News, Reddit r/programming이 초기 채널로 효과적이다 [확신도: 중간]

### 7. 전략 & 목표
- Notion, Linear가 AI 에이전트 전용 기능을 6개월 내에 추가하지 않을 것이다 [확신도: 낮음]

### 8. 팀
- 1인 PO가 PM 전략과 AI 에이전트 개발 감독을 동시에 수행할 수 있다 [확신도: 중간]
```

#### 실전 팁

> Teresa Torres가 말했듯, "좋은 팀은 아이디어의 4분의 3이 기대대로 동작하지 않을 것이라고 가정한다." 신규 제품에서는 특히 GTM과 팀 리스크를 과소평가하는 경향이 있다. 기술은 만들 수 있어도 "어떻게 고객을 만날 것인가"가 더 어려운 경우가 많다.

> 8가지 카테고리를 모두 채우려 하지 말고, 지금 가장 확신이 없는 영역에 집중하라. 기술 팀이라면 Value와 GTM이 가장 확신이 낮은 경우가 많다.

#### SprintX에서의 활용

SprintX는 기존 제품이지만, PO 가이드 Phase 0에서 "기능 우선(옵션 C)을 비채택한 이유"가 바로 이 스킬이 드러내는 리스크들이다. 특히 **GTM(채널이 없는 상태에서 어떻게 고객을 만날 것인가)**과 **팀(1인 PO + AI 에이전트라는 제약)** 리스크가 현재 SprintX의 핵심 위험 가정이다.

#### 관련 스킬

- **이전**: `brainstorm-ideas-new` (신규 아이디어 발산 후 위험 식별)
- **다음**: `prioritize-assumptions` (8가지 카테고리의 가정 중 어떤 것을 먼저 테스트할지), `brainstorm-experiments-new` (XYZ 가설로 가장 위험한 가정 테스트)
- **보완**: `identify-assumptions-existing` (기존 제품의 4가지 리스크로 빠르게 검토할 때), `swot-analysis` (외부 위험 요소 심화 분석)

### 7. 가정 우선순위화 (prioritize-assumptions)

**난이도**: 중급
**예상 소요 시간**: 20-30분
**슬래시 명령어**: `/pm-product-discovery:prioritize-assumptions`

#### 이 스킬은 언제 쓰나요?

- **시나리오 1**: 가정 목록이 너무 많아서 "어디서부터 테스트해야 하는가"를 결정해야 할 때
- **시나리오 2**: 어떤 실험을 먼저 할지 팀과 논쟁이 있을 때 -- 주관이 아닌 매트릭스로 합의하고 싶을 때
- **시나리오 3**: 제한된 시간과 자원으로 가장 효율적으로 발견을 진행하고 싶을 때

#### 실행 방법

```
/pm-product-discovery:prioritize-assumptions [제품명 + 가정 목록]
```

예시:
```
/pm-product-discovery:prioritize-assumptions SprintX - [identify-assumptions 산출물 붙여넣기]
```

Claude가 **임팩트 × 리스크 매트릭스**로 각 가정을 분류한다:

- **임팩트**: 이 가정이 검증되었을 때 가치 × 영향 받는 고객 수
- **리스크**: (1 - 확신도) × 노력

분류 결과:
| 임팩트 | 리스크 | 행동 |
|--------|--------|------|
| 높음 | 높음 | **지금 테스트하라** -- 가장 배울 것이 많다 |
| 높음 | 낮음 | **바로 구현하라** -- 위험 없이 높은 가치 |
| 낮음 | 낮음 | **나중에 처리하라** -- 급하지 않음 |
| 낮음 | 높음 | **포기하라** -- 투자 대비 가치가 없음 |

**입력으로 필요한 것**:
- `identify-assumptions-existing` 또는 `identify-assumptions-new` 산출물
- 각 가정의 영향 받는 고객 수 추정 (어려우면 상대적 크기로 대체)

#### 산출물 예시

```markdown
## 가정 우선순위 매트릭스: SprintX

| 가정 | 임팩트 | 리스크 | 분류 | 추천 실험 |
|------|--------|--------|------|---------|
| 개발자가 AI 결과를 검토하는 데 시간을 쓸 의향이 있다 | 높음 | 높음 | **지금 테스트** | Wizard of Oz: 5명에게 수동 검토 서비스 제공 후 재사용 의향 확인 |
| Goal-Task 계층이 직관적이다 | 높음 | 낮음 | **바로 구현** | 이미 구현됨. 추가 테스트 불필요 |
| Dark Mode가 전환율에 영향을 준다 | 낮음 | 높음 | **포기** | 전략적 Beachhead 세그먼트가 Dark Mode로 결정되지 않는 한 무시 |
| 이메일 다이제스트가 재방문율을 높인다 | 낮음 | 낮음 | **나중에** | Phase 3 이후 검토 |

### 이번 주 테스트할 가정 Top 3:
1. "개발자가 AI 결과를 검토하는 데 시간을 쓸 의향이 있다"
2. "Beachhead 세그먼트가 Hacker News에서 도달 가능하다"
3. "$5/월이 Beachhead 세그먼트의 지불 의향 범위 안에 있다"
```

#### 실전 팁

> 임팩트와 리스크를 정확한 숫자로 계산할 필요는 없다. "상대적으로 어떤 것이 더 임팩트가 크고 더 위험한가"를 팀이 합의하는 과정 자체가 가치 있다. 숫자보다 대화가 중요하다.

> ICE 점수(Impact × Confidence × Ease)나 RICE 공식을 원하면 `prioritize-features` 스킬이 더 적합하다. 이 스킬은 가정 자체를 우선순위화하는 것이고, 기능 백로그 우선순위화는 별개의 스킬이다.

#### SprintX에서의 활용

Phase 1 Step 1-6 산출물인 "가정 목록"을 이 스킬에 입력하면, **Phase 2에서 실행할 실험의 순서가 결정**된다. 특히:

- 이슈 #161-#167 실험의 실행 순서를 이 매트릭스로 결정한다
- "높은 임팩트, 높은 리스크" 가정이 먼저 테스트되어야 이후 #168-#185 개발 방향이 올바르게 잡힌다
- "포기"로 분류된 가정에 연결된 이슈는 Freeze 상태를 유지하면 된다

#### 관련 스킬

- **이전**: `identify-assumptions-existing` 또는 `identify-assumptions-new` (가정 목록 생성 후)
- **다음**: `brainstorm-experiments-existing` 또는 `brainstorm-experiments-new` (우선순위 Top 가정의 실험 설계)
- **보완**: `prioritize-features` (가정이 아닌 기능 백로그 우선순위화), `opportunity-solution-tree` (우선순위화된 가정을 OST에 통합)

### 8. 기능 우선순위화 (prioritize-features)

**난이도**: 중급
**예상 소요 시간**: 25-35분
**슬래시 명령어**: `/pm-product-discovery:prioritize-features`

#### 이 스킬은 언제 쓰나요?

- **시나리오 1**: 백로그에 기능이 너무 많아서 "이번 스프린트에 뭘 해야 하는가"를 결정해야 할 때
- **시나리오 2**: 모든 기능이 다 중요해 보여서 결정을 못 하고 있을 때 -- 프레임워크로 객관화하고 싶을 때
- **시나리오 3**: 이해관계자들이 각자 원하는 기능이 달라서 우선순위 합의가 필요할 때

#### 실행 방법

```
/pm-product-discovery:prioritize-features [제품명 + 기능 목록]
```

예시:
```
/pm-product-discovery:prioritize-features SprintX - 백로그 이슈 #168 Worker Daemon, #169 Provider Adapter, #170 Rate Limiting, #176 Velocity Chart, #181 AI 추천, #184 Dark Mode
```

Claude가 각 기능을 아래 4가지 기준으로 평가하고 Top 5를 선정한다:
- **임팩트**: 목표 결과를 얼마나 움직이는가? (가능하면 Opportunity Score = 중요도 × (1 - 현재 만족도) 적용)
- **노력**: 개발·디자인·조율에 얼마나 자원이 드는가?
- **리스크**: 얼마나 불확실성이 있는가? 어떤 가정이 필요한가?
- **전략 정렬**: 제품 비전과 현재 목표에 얼마나 부합하는가?

**입력으로 필요한 것**:
- 제품 목표와 성공 지표 (NSM, OKR)
- 기능 아이디어 목록
- 스프레드시트·백로그 파일 (있으면 직접 분석)

#### 산출물 예시

```markdown
## 기능 우선순위 평가: SprintX

**목표**: 첫 10명 유료 사용자 확보 (Q2 2026 OKR)

| 기능 | 임팩트 | 노력 | 리스크 | 전략 정렬 | 종합 |
|------|--------|------|--------|---------|------|
| #168 Worker Daemon | 높음 | 매우 높음 | 높음 | 높음 | 3위 |
| #184 Dark Mode | 낮음 | 낮음 | 낮음 | 낮음 | 제외 |
| #176 Velocity Chart | 중간 | 중간 | 낮음 | 중간 | 5위 |
| Onboarding 개선 | 높음 | 낮음 | 낮음 | 높음 | **1위** |

### Top 5 추천
1. **온보딩 개선** -- 첫 Task 생성까지 마찰 최소화. 낮은 노력, 높은 임팩트. NSM 직접 기여
2. **Activation 지표 추적** -- 무엇을 개선할지 알려면 먼저 측정해야 함
3. **#168 Worker Daemon** -- 핵심 차별화. 단, Beachhead가 AI 실행을 원한다는 것이 먼저 검증되어야 함
4. **#182 Activity Feed** -- 사용자 재방문 유도. 중간 노력, 중간 임팩트
5. **#170 Rate Limiting** -- 인프라 안정성. 유료 전환을 위한 신뢰도 확보

### 제외 근거
- **Dark Mode (#184)**: Beachhead 세그먼트의 구매 결정 요인이 아님. Phase 3 이후 재평가
```

#### 실전 팁

> 기능이 아니라 기회(문제)를 우선순위화하는 것이 올바른 순서다. "이 기능이 어떤 고객 문제를 해결하는가"에 답하지 못하는 기능은 상위에 올 수 없다. `opportunity-solution-tree`로 기회 공간을 먼저 정리한 후에 이 스킬을 사용하면 더 좋은 결과가 나온다.

> Kano 모델을 적용하면 "있어야 당연한 기능"(Basic), "있으면 좋은 기능"(Performance), "있으면 기쁜 기능"(Excitement)으로 나눌 수 있다. Excitement 기능이 임팩트가 가장 높아 보여도, Basic 기능이 없으면 이탈이 발생한다.

#### SprintX에서의 활용

PO 가이드 **Phase 2 Step 2-1 우선순위 프레임워크** 단계에서 핵심 스킬이다.

- **입력**: Phase 1의 모든 산출물(NSM, Beachhead, Value Prop) + 이슈 #168-#185 목록
- **목표**: 전략 결과 반영 전인 기존 P0-P3 라벨링을 전략 기반으로 재배열
- **핵심 질문**: "Worker Daemon(#168)이 정말 P0인가? Beachhead 세그먼트가 이것을 원하는가를 먼저 확인했는가?"
- RICE 공식을 쓰려면 Reach(영향 받는 사용자 수)를 추정해야 하는데, 인터뷰 결과가 이 추정의 근거가 된다

#### 관련 스킬

- **이전**: `prioritize-assumptions` (가정 검증 후 기능 선정), `analyze-feature-requests` (고객 요청 분석 후)
- **다음**: `create-prd` (우선순위 확정된 기능의 PRD 작성), `outcome-roadmap` (로드맵으로 배치)
- **보완**: `opportunity-solution-tree` (기회 공간 먼저 구조화), `brainstorm-okrs` (기능을 OKR에 매핑)

### 9. 기능 요청 분석 (analyze-feature-requests)

**난이도**: 중급
**예상 소요 시간**: 20-30분
**슬래시 명령어**: `/pm-product-discovery:analyze-feature-requests`

#### 이 스킬은 언제 쓰나요?

- **시나리오 1**: 사용자나 이해관계자로부터 기능 요청이 쌓여 있는데 어떻게 처리할지 모를 때
- **시나리오 2**: "기능 요청을 다 들어줄 수는 없는데, 어떤 것을 선택해야 하는가"에 답이 필요할 때
- **시나리오 3**: 인터뷰, 설문, 지원 티켓, GitHub 이슈 등 다양한 채널에서 모인 요청을 통합 분석할 때

#### 실행 방법

```
/pm-product-discovery:analyze-feature-requests [제품명 + 기능 요청 목록]
```

예시:
```
/pm-product-discovery:analyze-feature-requests SprintX - [인터뷰 메모와 GitHub 이슈 목록 붙여넣기]
```

Claude가 단계적으로 진행한다:
1. 요청들을 **테마별로 묶어** 이름을 붙인다 (예: "알림 관련", "AI 실행 관련", "시각화 관련")
2. 각 테마의 **전략 정렬도** 평가
3. 임팩트·노력·리스크·전략 정렬 기준으로 **Top 3 기능** 선정
4. 각 Top 3에 대해 **대안 솔루션**, **위험 가정**, **최소 노력 테스트 방법** 제시

**핵심 원칙**: "고객이 솔루션을 디자인하게 두지 말라." 고객이 "슬랙 연동을 해주세요"라고 하면, 실제 문제는 "결과를 빠르게 확인하고 싶다"일 수 있다. 요청 뒤에 있는 기회(문제)를 찾는 것이 이 스킬의 목적이다.

**입력으로 필요한 것**:
- 기능 요청 목록 (텍스트, 스프레드시트, CSV 형식 가능)
- 제품 목표와 우선순위 (어떤 목표를 기준으로 평가할 것인가)

#### 산출물 예시

```markdown
## 기능 요청 분석: SprintX

**평가 목표**: 첫 10명 유료 사용자의 활성화율 향상

### 테마 분류
- **알림/커뮤니케이션** (6건): Slack 연동, 이메일 다이제스트, 모바일 푸시
- **AI 실행 시각화** (4건): 실행 로그 보기, 진행률 표시, 에러 요약
- **협업** (3건): 팀 댓글, 멘션, 공유 링크
- **플랫폼/UI** (3건): Dark Mode, 키보드 단축키, 모바일 앱

### Top 3 기능 선정

#### 1위: AI 실행 로그 시각화 (AI 실행 시각화 테마)
- **고객 문제**: "AI가 무엇을 하고 있는지 모르겠다"
- **왜 선정됐는가**: 4건의 요청 + Beachhead 세그먼트의 핵심 불안 해소
- **대안 솔루션**: 실시간 로그 vs 완료 요약 vs 진행률 바
- **위험 가정**: Worker Daemon이 실행 로그를 생성한다는 전제 (현재 미구현)
- **최소 테스트**: Wizard of Oz -- 수동으로 로그를 입력해 화면에 표시

#### 2위: Slack 연동 (알림 테마)
- **고객 문제**: "결과를 확인하러 SprintX를 매번 열기 귀찮다"
- **대안**: Slack 연동 vs 이메일 다이제스트 (더 쉬움)
- **위험 가정**: 사용자가 Slack을 쓴다는 가정 -- 인터뷰에서 확인 필요

#### 3위: 에러 요약 / 반려 이유 기록 (AI 실행 시각화 테마)
- ...

### 제외된 요청
- **Dark Mode**: "좋으면 좋겠다" 수준의 Nice-to-have. 활성화율에 영향 없음
- **모바일 앱**: 개발 노력이 너무 크고, Beachhead(개발자)는 데스크탑 우선
```

#### 실전 팁

> "고객이 가장 많이 요청한 기능"이 "가장 중요한 기능"이 아닐 수 있다. 10명이 요청해도 아무도 실제로 쓰지 않을 수 있고, 1명만 요청해도 이탈을 막는 핵심 기능일 수 있다. 요청 수보다 "이것이 없어서 못 쓰겠다"는 강도를 더 중요하게 보라.

> 기능 요청을 분석할 때 Opportunity Score(중요도 × (1 - 현재 만족도))를 적용하면 더 정확하다. "이 문제가 얼마나 중요한가? 현재 솔루션에 얼마나 만족하는가?"를 인터뷰에서 물어보면 계산할 수 있다.

#### SprintX에서의 활용

Phase 3(격주 반복 운영)에서 매 사이클마다 사용자 인터뷰와 Waitlist 등록자 피드백을 이 스킬로 처리한다. Phase 2에서 실험을 시작한 후 피드백이 쌓이기 시작하면, 이 스킬이 백로그 재정렬의 주요 입력이 된다.

- **입력 채널**: 사용자 인터뷰 노트, GitHub 이슈, Waitlist 등록자 이메일 피드백
- **산출물 연결**: Top 3 → `prioritize-features` → `create-prd` → 개발 핸드오프

#### 관련 스킬

- **이전**: `summarize-interview` (인터뷰 노트를 요약해서 이 스킬에 입력)
- **다음**: `prioritize-features` (Top 3를 다른 백로그 항목과 비교해 최종 순위 결정), `identify-assumptions-existing` (선정된 기능의 위험 가정 심화 분석)
- **보완**: `opportunity-solution-tree` (테마를 OST의 기회 레벨로 매핑), `customer-journey-map` (어느 단계에서 요청이 나오는지 파악)

### 10. 기회-솔루션 트리 (opportunity-solution-tree)

**난이도**: 고급
**예상 소요 시간**: 40-60분
**슬래시 명령어**: `/pm-product-discovery:opportunity-solution-tree`

#### 이 스킬은 언제 쓰나요?

- **시나리오 1**: 발견 과정이 산만해질 때 -- "인터뷰도 하고 실험도 하는데, 이게 어떻게 연결되는지 모르겠다"
- **시나리오 2**: "무엇을 만들지"보다 "어떤 고객 문제를 해결하고 있는가"에 집중하고 싶을 때
- **시나리오 3**: 팀 전체가 같은 발견 구조를 공유하고 싶을 때 -- 발견의 "지도"가 필요할 때

#### 실행 방법

```
/pm-product-discovery:opportunity-solution-tree [제품명 또는 목표 결과]
```

예시:
```
/pm-product-discovery:opportunity-solution-tree SprintX - "7일 유지율 20% 달성"
```

**OST 4단계 구조** (Teresa Torres, *Continuous Discovery Habits*):

```
목표 결과 (Desired Outcome)
    └── 기회 1: 사용자가 첫 Task 생성 후 가치를 느끼지 못한다
    │       └── 솔루션 1a: 온보딩 체크리스트
    │       └── 솔루션 1b: 예시 데이터 자동 생성
    │       └── 솔루션 1c: 첫 Task AI 자동 완성
    │               └── 실험: Feature stub으로 클릭율 측정
    └── 기회 2: AI 실행 결과를 확인할 방법이 없다
    │       └── 솔루션 2a: 실행 로그 타임라인
    │       └── 솔루션 2b: 요약 알림
    │               └── 실험: Wizard of Oz (수동 로그 작성)
    └── 기회 3: ...
```

**핵심 원칙**:
- 한 번에 하나의 목표 결과에 집중
- 기회는 고객의 언어로 ("나는 ... 하기가 힘들다")
- 솔루션은 하나만 선택하지 말고 3개 이상 비교
- 발견은 선형이 아니다 -- 실험이 실패하면 다른 솔루션으로 이동

**입력으로 필요한 것**:
- 측정 가능한 목표 결과 (OKR이나 NSM)
- 사용자 연구 데이터 (인터뷰 노트, 설문, 분석 데이터)
- 기존에 식별된 기회나 솔루션 아이디어 (있으면)

#### 산출물 예시

```markdown
## Opportunity Solution Tree: SprintX

### 목표 결과
"7일 활성 사용자 중 Task를 3개 이상 완료한 비율을 20%로 높인다" (현재: 측정 전)

### 기회 공간 (3-7개)
1. "처음 접속했을 때 무엇을 해야 할지 모르겠다" [기회 점수: 높음]
2. "AI가 무엇을 하고 있는지 알 수 없다" [기회 점수: 높음]
3. "Task 완료 후 다음 단계가 자연스럽게 이어지지 않는다" [기회 점수: 중간]

### 기회 1 심화: "처음 접속했을 때 무엇을 해야 할지 모르겠다"

**솔루션 후보 3개**:
- 솔루션 A: 온보딩 체크리스트 (낮은 노력, 중간 임팩트)
- 솔루션 B: 예시 프로젝트 자동 생성 (중간 노력, 높은 임팩트)
- 솔루션 C: 인터랙티브 투어 (높은 노력, 높은 임팩트)

**솔루션 A 실험**:
- 가정: 체크리스트를 보면 다음 단계를 따라간다
- 실험: Feature stub으로 체크리스트 UI만 먼저 추가
- 지표: 7일 내 체크리스트 3단계 완료율
- 성공 기준: 체험 사용자 중 30% 이상 완료

### 우선순위 결정
기회 점수(중요도 × (1 - 현재 만족도)) 기준으로 기회 1, 2를 우선 탐색
```

#### 실전 팁

> OST를 처음 만들 때 완벽하게 채우려 하지 말라. 처음에는 가장 확신이 높은 기회 1-2개만 깊게 파고, 나머지는 잠재 기회로 표시해 두면 된다. Teresa Torres는 OST를 "매주 업데이트하는 살아있는 문서"로 사용하라고 했다.

> 솔루션을 하나만 생각하는 "첫 아이디어 함정"을 피하려면, 반드시 3개 이상의 솔루션을 나란히 비교해야 한다. 다양한 솔루션을 비교해야 "왜 이 솔루션인가"에 답할 수 있다.

#### SprintX에서의 활용

PO 가이드 전체를 하나의 OST로 볼 수 있다. Teresa Torres의 Continuous Discovery가 SprintX PO 가이드의 이론적 기반이다.

- **목표 결과**: Phase 2 Step 2-2에서 설정한 OKR의 Key Result (예: "유료 전환 10명")
- **기회 공간**: Phase 1 인터뷰 + 고객 여정 맵에서 식별된 끊김 지점
- **솔루션**: 이슈 #168-#185 중 각 기회에 연결된 것들
- **실험**: 이슈 #161-#167의 실험들
- 이 구조가 명확해지면 25개 이슈 중 어떤 것이 "실험"이고 어떤 것이 "솔루션 구현"인지 구분이 된다

#### 관련 스킬

- **이전**: `brainstorm-ideas-existing` (발산된 아이디어를 OST 솔루션으로), `summarize-interview` (기회 공간 채우기), `north-star-metric` (목표 결과 정의)
- **다음**: `brainstorm-experiments-existing` (각 솔루션의 실험 설계), `prioritize-features` (솔루션을 백로그로 정렬)
- **보완**: `customer-journey-map` (기회 공간 식별에 도움), `metrics-dashboard` (목표 결과 측정 방법)

### 11. 인터뷰 스크립트 (interview-script)

**난이도**: 초급
**예상 소요 시간**: 15-20분
**슬래시 명령어**: `/pm-product-discovery:interview-script`

#### 이 스킬은 언제 쓰나요?

- **시나리오 1**: 사용자 인터뷰를 처음 해보는데 무엇을 물어봐야 할지 모를 때
- **시나리오 2**: 인터뷰를 해봤지만 매번 다른 정보를 얻게 되어 비교하기 어려울 때 -- 구조화된 가이드가 필요할 때
- **시나리오 3**: 특정 가설이나 기능 아이디어를 검증하기 위한 맞춤형 질문지가 필요할 때

#### 실행 방법

```
/pm-product-discovery:interview-script [연구 주제 또는 검증하려는 가정]
```

예시:
```
/pm-product-discovery:interview-script SprintX - AI 코딩 에이전트를 사용하는 개발자의 프로젝트 관리 방식과 불편함 탐색
```

Claude가 구조화된 인터뷰 스크립트를 생성한다:
- **여는 말** (2-3분): 소개, 목적, 기대치 설정
- **워밍업** (5분): 배경, 역할, 일반적인 상황
- **핵심 탐색** (15-20분): JTBD 기반 질문 (현재 상황, 고통, 원하는 결과, 지불 의향)
- **프로빙 기법**: "좀 더 말씀해 주시겠어요", "왜 그런가요?", "구체적인 예를 들어주시겠어요?"
- **마무리** (3-5분): 놓친 주제 확인, 소개 요청

**The Mom Test 원칙** 적용:
- "이 기능을 쓰실 것 같아요?"가 아니라 "마지막으로 이걸 한 게 언제인가요?" -- 과거 행동을 물어라
- 제품을 피칭하지 말고 그들의 삶을 이해하라
- "그거 좋아 보이네요"는 아무 정보도 아니다 -- 칭찬은 노이즈다
- 강한 감정이 나타나는 곳이 진짜 고통이다

노트 기록 템플릿도 함께 제공된다:
```
참가자: [이름/ID]     날짜:
핵심 Jobs: [그들이 하려는 것]     현재 솔루션: [지금 쓰는 것]
가장 큰 고통: [#1 불편함]     원하는 결과: [성공 기준]
지불 의향: [투자하는 시간/돈]     예상치 못한 발견: []
```

**입력으로 필요한 것**:
- 인터뷰 목적과 검증하려는 가정
- 타겟 인터뷰 대상 설명
- 페르소나, 가설 목록, 이전 인터뷰 노트 (있으면)

#### 산출물 예시

```markdown
## 인터뷰 스크립트: SprintX - AI 코딩 에이전트 사용자

### 여는 말
"안녕하세요. 저는 스프린트 관리 도구를 만들고 있는데요,
오늘은 제 도구를 소개하는 것이 아니라 여러분이 일하는 방식을 이해하고 싶어서요.
맞는 답도 틀린 답도 없습니다. 30분 정도 시간 내주셔서 감사합니다."

### 핵심 질문 (선택 5-7개)
1. "AI 코딩 에이전트를 마지막으로 쓴 게 언제인가요? 어떤 작업이었나요?"
2. "그 결과물을 어떻게 확인하셨나요? 어디에 기록하셨나요?"
3. "AI가 잘못된 코드를 만들었을 때 어떻게 하셨나요?"
4. "지금 프로젝트 진행 상황을 어떻게 관리하시나요?"
5. "그 부분에서 가장 힘든 게 뭔가요?"
6. "이 문제를 해결하려고 다른 방법을 써보셨나요? 어떻게 됐나요?"
```

#### 실전 팁

> 인터뷰 중 "왜요?"를 3번 물어보라. 처음 대답은 표면적인 이유고, 세 번째 "왜?"에서 진짜 동기가 나온다. 예: "저는 Jira가 너무 복잡해요" → "왜 복잡하게 느끼시나요?" → "팀 협업 기능이 저에게는 필요 없어서요" → "혼자 쓰시는 건가요? 그럼 어떤 기능이 가장 필요하세요?"

> 인터뷰는 가능하면 PM 혼자 하지 말고 Product Trio(PM + 디자이너 + 엔지니어) 함께 참석하는 것이 Teresa Torres의 권고다. 1인 PO라면 인터뷰 녹음 후 `summarize-interview`로 AI가 분석하도록 하면 보완이 된다.

#### SprintX에서의 활용

PO 가이드 **Phase 1 Week 1 병행** 활동에서 사용하는 스킬이다. Week 1에 3명 인터뷰 목표를 세우기 전에 먼저 이 스크립트를 만들어 두어야 한다.

- **타겟**: Waitlist 등록자, Reddit/Hacker News의 AI 코딩 에이전트 사용자, PO 개인 네트워크
- **검증 가정 포함**: Step 1-3(경쟁 분석)에서 도출된 "SprintX가 해결하는 핵심 문제"를 중심으로 질문 구성
- **주의**: "SprintX 좋지 않나요?"처럼 유도 질문은 절대 금지. 기능 아이디어 묻지 말고 문제 묻기
- 완성된 스크립트는 `docs/product/INTERVIEW_SCRIPT.md`에 저장 권장

#### 관련 스킬

- **이전**: `identify-assumptions-existing` 또는 `identify-assumptions-new` (검증하려는 가정을 질문으로 변환)
- **다음**: `summarize-interview` (인터뷰 완료 후 즉시 사용), `brainstorm-ideas-existing` (인터뷰 인사이트를 기반으로 아이디어 발산)
- **보완**: `value-proposition` (인터뷰에서 얻은 JTBD를 가치 제안으로 연결), `analyze-feature-requests` (여러 인터뷰에서 나온 요청을 통합 분석)

### 12. 인터뷰 요약 (summarize-interview)

**난이도**: 중급
**예상 소요 시간**: 10-15분 (전문이 있을 경우)
**슬래시 명령어**: `/pm-product-discovery:summarize-interview`

#### 이 스킬은 언제 쓰나요?

- **시나리오 1**: 인터뷰 녹음을 텍스트로 변환했는데, 이 긴 텍스트에서 핵심을 뽑아야 할 때
- **시나리오 2**: 인터뷰마다 얻는 정보가 다른 형식이라 비교가 어려울 때 -- 표준화된 요약이 필요할 때
- **시나리오 3**: 팀(또는 AI 에이전트)에게 인터뷰 결과를 공유할 때 -- 전문 50페이지가 아닌 1페이지 요약 필요

#### 실행 방법

```
/pm-product-discovery:summarize-interview [인터뷰 전문 붙여넣기 또는 파일 첨부]
```

예시:
```
/pm-product-discovery:summarize-interview
[인터뷰 녹취 전문 여기에 붙여넣기]
```

Claude가 아래 템플릿으로 요약을 작성한다:

```
날짜: [인터뷰 일시]
참가자: [이름/역할]
배경: [고객 배경 정보]

현재 솔루션: [지금 사용하는 도구/방법]

현재 솔루션에서 좋은 점:
- [JTBD, 원하는 결과, 중요도, 만족도]

현재 솔루션의 문제:
- [JTBD, 원하는 결과, 중요도, 만족도]

핵심 인사이트:
- [예상치 못한 발견 또는 주목할 만한 인용]

액션 아이템:
```

**입력으로 필요한 것**:
- 인터뷰 녹취 전문 (텍스트, PDF, 오디오 텍스트 변환 파일)

```markdown
## 인터뷰 요약: 홍길동 (프리랜서 개발자)

**날짜**: 2026-03-11 14:00
**참가자**: 홍길동, 프리랜서 개발자 3년차, Claude Code 주 5회 사용
**배경**: 사이드 프로젝트 2개 + 프리랜서 클라이언트 프로젝트 3개를 동시 관리

**현재 솔루션**: Notion + GitHub Issues + 카카오톡 자기 자신에게 메모

**현재 솔루션에서 좋은 점**:
- Notion: 문서 정리 (중요도: 높음, 만족도: 중간)

**현재 솔루션의 문제**:
- AI가 만든 코드를 Notion에 수동으로 복사해야 함 (중요도: 높음, 만족도: 낮음)
- 어떤 AI 작업이 완료됐는지 흩어져 있어 추적 불가 (중요도: 높음, 만족도: 매우 낮음)

**핵심 인사이트**:
- "AI가 일을 다 했는데, 나는 확인하러 다시 돌아와서 내가 뭘 시켰는지 찾아야 해요"
- 지불 의향 언급: "이게 자동으로 되면 월 10달러도 낼 것 같아요"

**액션 아이템**:
- 2026-03-18, PO, 5명 더 인터뷰해서 "흩어진 AI 작업 추적" 패턴 반복 여부 확인
- 2026-03-15, PO, 인터뷰 결과를 brainstorm-ideas-existing 스킬에 입력
```

#### 실전 팁

> "초등학생도 이해할 수 있는 언어"로 쓰는 것이 이 스킬의 지침이다. 전문 용어를 쓰지 말고 고객의 실제 언어를 그대로 사용하자. 가장 강력한 인용문은 그대로 따옴표로 보존해 두어라. 나중에 랜딩 페이지 카피로 사용할 수 있다.

> 인터뷰 직후 24시간 안에 요약하라. 기억이 흐릿해지기 전에 처리해야 "핵심 인사이트"가 정확하다. 1주일 후에 하면 녹취 전문이 없으면 세부 내용을 잃는다.

#### SprintX에서의 활용

PO 가이드 **Phase 1 Week 1 병행 인터뷰** + **Phase 3 매주 인터뷰**에서 매번 사용하는 스킬이다.

- **즉시 활용**: 인터뷰 직후 녹취 전문을 이 스킬에 넣으면 `docs/product/INTERVIEW_[날짜]_[이름].md`로 저장
- **패턴 발견**: 3명 이상 인터뷰 요약이 쌓이면 `analyze-feature-requests`로 통합 분석
- **JTBD 정제**: 요약에서 반복되는 JTBD 패턴이 나오면 `value-proposition` 스킬의 "Who/Why" 섹션을 업데이트
- **지불 의향 신호**: 인터뷰에서 가격 언급이 나오면 Phase 1 Step 1-5(가격 전략)의 중요한 입력이 됨

#### 관련 스킬

- **이전**: `interview-script` (인터뷰 진행 후 즉시 사용)
- **다음**: `brainstorm-ideas-existing` (인터뷰 인사이트로 아이디어 발산), `analyze-feature-requests` (여러 인터뷰 통합 분석), `opportunity-solution-tree` (기회 공간 업데이트)
- **보완**: `value-proposition` (JTBD를 가치 제안에 반영), `identify-assumptions-existing` (인터뷰에서 발견된 새로운 위험 가정 추가)

### 13. 지표 대시보드 설계 (metrics-dashboard)

**난이도**: 고급
**예상 소요 시간**: 40-60분
**슬래시 명령어**: `/pm-product-discovery:metrics-dashboard`

#### 이 스킬은 언제 쓰나요?

- **시나리오 1**: "우리 제품이 잘 되고 있는지"를 매주 확인하는 체계를 만들고 싶을 때
- **시나리오 2**: 지표가 너무 많아서 "어떤 것을 매일/매주/매월 봐야 하는가"가 혼란스러울 때
- **시나리오 3**: GA나 Amplitude 대시보드를 어떻게 구성해야 할지 모를 때 -- 어떤 지표를 어떤 형태로 볼지 설계가 필요할 때

#### 실행 방법

```
/pm-product-discovery:metrics-dashboard [제품명 + 현재 OKR 또는 NSM]
```

예시:
```
/pm-product-discovery:metrics-dashboard SprintX - North Star Metric: "주간 활성 사용자 중 Task를 완료한 비율", OKR: Q2 2026 첫 10명 유료 전환
```

Claude가 4계층 지표 체계를 설계한다:

1. **North Star Metric**: 핵심 가치 전달을 측정하는 단 하나의 지표
2. **Input Metrics (3-5개)**: NSM을 움직이는 레버 지표
3. **Health Metrics**: 제품 전체 건강을 확인하는 가드레일
4. **Business Metrics**: 매출, 비용, 단위 경제학

각 지표마다 정의·데이터 출처·시각화 형태·목표값·알림 임계값이 정의된다.

**좋은 지표의 4가지 기준** (Ben Yoskovitz, *Lean Analytics*):
1. 이해하기 쉽다 -- 팀 공통 언어가 된다
2. 비교 가능하다 -- 지난주 대비, 지난달 대비
3. 비율 또는 속도다 -- 절대값보다 훨씬 유용하다
4. 행동을 바꾼다 -- "이 지표가 나빠도 우리가 아무것도 안 할 것 같다면, 그것은 나쁜 지표다"

**입력으로 필요한 것**:
- 현재 NSM 또는 주요 비즈니스 목표
- OKR 또는 분기 목표
- 기존 분석 도구와 데이터 현황
- 전략 문서 파일 (있으면)

#### 산출물 예시

```markdown
## 제품 지표 대시보드 설계: SprintX

### North Star Metric
**"7일 내 Task를 3개 이상 완료한 신규 가입자 비율"**
- 정의: 가입일로부터 7일 내 3개 이상 Task를 완료 상태로 바꾼 사용자 수 / 전체 신규 가입자 수
- 데이터 출처: Supabase tasks 테이블 (status = 'done', created_at 기준)
- 시각화: 주간 선 차트
- 목표: 20% (현재: 측정 전)
- 알림: 10% 미만이면 즉시 알림

### Input Metrics (NSM을 움직이는 레버)
| 지표 | 정의 | 목표 | 알림 |
|------|------|------|------|
| 온보딩 완료율 | project_created / signup 완료자 비율 | 60% | 40% 미만 |
| 첫 Goal 생성율 | first_goal_created / dashboard_entered 비율 | 50% | 30% 미만 |
| Task 생성 → 완료 전환율 | 완료된 Task / 전체 Task 비율 | 40% | 20% 미만 |

### Health Metrics
- 에러율 (API 5xx): 0.1% 미만 목표
- 페이지 로드 시간 (p95): 2초 미만

### Business Metrics
- MRR (월 반복 매출): 현재 $0. 목표 $500 (10명 × $5/월)
- Churn Rate: 월 5% 미만
- CAC (고객 획득 비용): 추정 미정 (GTM 채널 결정 후 계산)

### 대시보드 레이아웃

┌─────────────────────────────────────────────┐
│  NSM: 7일 Task 3개 완료율  [현재: 측정 전]  │
│  Trend: --                                   │
├──────────────────┬──────────────────────────┤
│  온보딩 완료율   │  첫 Goal 생성율          │
│  [목표: 60%]    │  [목표: 50%]             │
├──────────────────┼──────────────────────────┤
│  Task 완료 전환율│  신규 가입자 수          │
│  [목표: 40%]    │  [주간 선 차트]          │
├──────────────────┴──────────────────────────┤
│  HEALTH: [에러율] [p95 로드 시간]           │
├─────────────────────────────────────────────┤
│  BUSINESS: [MRR] [Churn] [유료 전환수]      │
└─────────────────────────────────────────────┘

### 리뷰 주기
- 매일: 에러율, 가입자 수 (이상 감지)
- 매주: Input Metrics, NSM 트렌드
- 매월: Business Metrics, NSM 목표 대비 진행률
- 분기별: 지표 체계 재검토 (좋은 지표인가?)

### 추천 도구
- PostHog: 프로덕트 분석 (오픈소스, 무료 시작 가능)
- Metabase: Supabase 연결 SQL 대시보드
- Plausible / GA4: 랜딩 페이지 웹 분석 (analytics.ts 이미 구현됨)
```

#### 실전 팁

> "허영 지표(Vanity Metrics)"를 조심하라. 총 가입자 수, 페이지뷰 같은 숫자는 보기 좋지만 행동을 바꾸지 않는다. "이 숫자가 나쁘면 우리는 무엇을 다르게 할 것인가?"를 물어봐서 대답이 없으면 그것은 나쁜 지표다.

> 지표는 Leading(선행 지표)과 Lagging(후행 지표)으로 나뉜다. 매출은 후행 지표고, 활성화율은 선행 지표다. 선행 지표를 개선해야 나중에 매출이 따라온다. 초기 스타트업은 선행 지표에 집중하라.

#### SprintX에서의 활용

PO 가이드 **Phase 1 Step 1-4 North Star Metric** 정의 이후에 사용한다. NSM이 확정되면 그것을 중심으로 대시보드를 설계해야 한다.

- **즉시 활용 가능한 데이터**: Phase 0에서 가동한 GA/Plausible과 `src/lib/analytics.ts`의 13개 이벤트
- **SQL 활용**: `scripts/onboarding-funnel.sql`의 8단계 퍼널이 Input Metrics의 기초
- **Supabase 연결**: Metabase를 Supabase에 연결하면 비용 없이 SQL 기반 대시보드를 만들 수 있음
- 이슈 #166(퍼널 측정)과 직결됨 -- 이 스킬의 산출물이 #166 이슈의 기술 요구사항이 됨

#### 관련 스킬

- **이전**: `north-star-metric` (NSM 정의 후 대시보드 설계), `opportunity-solution-tree` (목표 결과를 지표로 변환)
- **다음**: `brainstorm-okrs` (지표 목표를 OKR의 Key Result로), `outcome-roadmap` (지표 목표를 로드맵에 반영)
- **보완**: `brainstorm-experiments-existing` (지표가 낮을 때 어떤 실험을 할지), `customer-journey-map` (퍼널 지표의 각 단계 이해)
